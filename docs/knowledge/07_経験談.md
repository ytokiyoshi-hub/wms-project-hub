## このドキュメントの目的

WMS仕様策定の「過程」と「判断の理由」を記録する。  
コード・仕様書には残らない「なぜそう決めたか」の文脈を保存し、  
後から関わるメンバーが設計意図を理解できるようにする。

> **記録原則**：「何を決めたか」ではなく「なぜそう決めたか」「何を捨てたか」を書く

---

## 1. このWMSが生まれた背景

### プロジェクトの起点

「物流のプロ（非エンジニア）」の顧客が抱えていた課題：  
「3PL対応の倉庫管理システムが市場にない、あっても使いにくい、  
または3PLの実態に合っていない機能で溢れている」

既存のWMSに対する不満：
- 単一荷主向けに設計されていて複数荷主対応が後付けで弱い
- 請求書の計算根拠を荷主に提示する機能が貧弱
- 人員管理が別システムになっていてコスト計算ができない

→ **「ゼロから3PL専業でWMSを仕様設計する」** というプロジェクトが始まった。

---

## 2. 仕様策定の進め方

### 顧客（物流プロ）+ Claude（別チャット）の議論

仕様策定は、物流の専門知識を持つ顧客がClaudeとの対話を通じて進めている。  
「こんな業務があって、こういう課題がある」→ 「では機能としてはこうなる」という対話型設計。

**この方法の強み：**  
- 物流の実態から外れた机上の空論になりにくい
- 専門家の暗黙知がシステム要件として言語化される
- 「言語化されていなかった課題」が議論の中で発掘される

**この方法の弱み：**  
- 議論の記録がチャットに散在する（→ CLAUDE.mdに集約中）
- 設計判断の背景が後から追いにくい（→ このドキュメントで補完）

## 3. バージョン変遷と主要な設計転換

### v1.0 → v2.x：3PL対応の本格化

**転換点：**  
「このWMSは何台の倉庫で使うのか？」という問いへの答えが  
「1社専用」から「複数荷主の3PL専業」に確定した時点。

**影響：**  
マスタ設計・在庫管理・請求管理のすべてが「荷主別分離」を前提に再設計された。  
単なる在庫管理ツールから「3PLビジネスの基盤システム」への転換。

### v2.x → v3.0：帳票・ER図の整備

- 印刷帳票仕様書（v3.0）を整備
- ER図（v3.0）でデータ構造を可視化
- 75画面のモック整備が完了

### v3.0 → v3.1：新4カテゴリの追加（2026-05-02〜04）

**P・Q・R・Sの4カテゴリ、38機能の追加**

| カテゴリ | 名称 | 背景 |
|---------|------|------|
| P | ロケ種別による引当・補充最適化 | 「在庫が枯渇してもピッキングを続けるには補充の自動化が必要」 |
| Q | 配送業者の自動振り分け | 「手動の配送業者振り分けが毎日のボトルネック」 |
| R | 商品コード体系 | 「荷主のコードと内部コードの管理が属人化している」 |
| S | 梱包ロジック・梱包材管理 | 「梱包材のコストが見えない・最適化できていない」 |

## 4. 重要な設計判断の記録

### 判断記録1：「その他出庫」の廃止

**経緯：**  
初期仕様では「廃棄出庫」「サンプル出庫」「仕入先返品出庫」を  
受注とは独立した「その他出庫」機能として検討していた。

**工程14レビューでの発見：**  
顧客から「倉庫業者は荷主の指示なしに在庫を動かせない」という指摘。  
法律（倉庫業法）・契約上の問題に加え、荷主への実績報告でも矛盾が生じることが判明。

**決定：**  
「受注を伴わない出庫」という概念を廃止。  
廃棄・サンプル・返品もすべて「荷主の出荷指示」として処理する。  
W-310・HT-801 画面を廃止確定。

**教訓：**  
「物流の常識」がシステム設計の前提になる。  
業界知識のない設計者が作ると、法律・契約に違反する機能が生まれるリスクがある。

---

### 判断記録2：ピッキング担当の「専任制」廃止

**検討していたこと：**  
「Aゾーン担当」「Bゾーン担当」のように作業員を固定化する設計。  
作業員が担当エリアを熟知して効率化できるという仮説。

**廃止の理由：**  
- 少人数倉庫（3〜5人）では担当固定が機能しない（欠員時に詰む）
- 実際の倉庫では「今日の物量に合わせてリーダーが配置を決める」運用が現実的
- 物量の多いエリアに人を集中させる柔軟性が失われる

**確定設計：**  
E-2「作業者は固定の専門担当を持たない」  
E-3「日次でリーダーが柔軟に担当エリアを割り振る」

---

### 判断記録3：引当方式の「受注ごと選択」

**背景：**  
「エリア並行」と「ゾーンリレー」のどちらが良いかという議論。

**検討の結果：**  
どちらが優れているかは出荷内容によって異なる：
- 大量・単品中心 → エリア並行が効率的
- 多品目・少量 → ゾーンリレーが効率的

**確定設計：**  
E-1「受注ごとに引当方式を選択（エリア並行/ゾーンリレー）」  
「どちらかに統一する」ではなく「使い分けられる」仕様が正解だった。

---

### 判断記録4：枝番管理の「履歴重視」

**背景：**  
荷主の商品コードが変更（リニューアル・規格変更）された場合の管理方針。

**選択肢：**  
1. 古いコードを新しいコードに書き換える（シンプルだが履歴が消える）
2. 古いコードは残し、新しいコードは枝番-02として追加（複雑だが履歴が残る）

**決定：**  
方式2「履歴重視」を採用（R-4）。  
理由：倉庫では「いつのロットが何個ある」という在庫履歴が重要で、コードを書き換えると過去データとの照合ができなくなる。

## 5. 「物流の当たり前」がシステムに与える影響

仕様策定を通じて発見した「物流の当たり前がシステム設計に与える影響」のまとめ。

### 5-1 倉庫業者は「保管の委託を受けた代理人」

→ 荷主の指示なしに在庫を動かせない  
→ 「その他出庫」という概念が存在しえない  
→ すべての在庫移動は「出荷指示」として処理する

### 5-2 FIFOは「運用の話」ではなく「食品の法令レベルの要件」

→ 消費期限管理がある食品では、古いロットから必ず出さなければならない  
→ システムが強制しないと「うっかりFIFO違反」が発生する  
→ E-6「ピッキングはFIFO強制」が必須要件

### 5-3 保管料計算は「業界慣行」を反映しなければ使えない

→ 3期制・2期制・坪貸しは業界標準の計算方式  
→ 独自の計算方式にしても荷主に説明できない  
→ 業界標準方式をそのまま実装することが最も合理的

### 5-4 「在庫が正確であること」への信頼が3PLビジネスの根幹

→ 在庫が合わない = 荷主からの信頼失墜 = 取引終了  
→ HTによるスキャン確認・ダブルチェックの仕組みが重要  
→ 棚卸し差異の即時報告機能が必須

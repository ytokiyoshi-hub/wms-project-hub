# アドバイザー（今井先生）連携フロー

> 起票：2026-05-08 / Phase 8-H（まーちゃん）
> 今井先生（プログラミング講師・テクニカルアドバイザー兼設計ゲートキーパー）との連携運用ルール。

---

## 1. 役割定義（時吉さん最終承認待ち）

**今井先生 = テクニカルアドバイザー兼設計ゲートキーパー**

| 項目 | 内容 |
|------|------|
| 主タスク | 致命傷ライン15項目の妥当性レビュー / ER 図レビュー / 実装着手前の設計承認 |
| 副タスク | 「9割突っ走り」と「完璧詰め」の境界線判断 / トラブル時のスポット相談 |
| やらないこと | 実装（コード書き）／ 雑務 ／ AI 大量生産物の全数チェック |
| 関与時間 | 週 5〜10 時間（時吉さん想定・拡張余地あり） |
| 立場 | 助言者（最終判断は時吉さん） |

> ⚠️ 役割定義は時吉さんと先生の合意で確定する。本ファイルは PM 提案の叩き台。

## 2. 連絡方法

### 2-1. 既存 LINE（時吉さん⇄先生・個人 LINE）

- **連絡経路：時吉さん経由**
  - 先生への提案・質問 → まーちゃんが時吉さんに伝える → 時吉さんが LINE で先生に投げる
  - 先生からの回答 → 時吉さん経由でまーちゃんに伝わる
- **理由**：先生の個人 LINE を AI が直接使うのは現状不適切。時吉さんが翻訳役になる方が情報精度が高い。
- **将来**：先生用 LINE Bot（まーちゃん用とは別 Channel）を希望されれば構築可能。

### 2-2. GitHub（情報共有・非同期）

- **共有経路：ytokiyoshi-hub の repo（先生に閲覧権限付与済）**
- **使い方**：
  - 仕様書・判断シート・ER 図 = `specs/` ディレクトリ
  - 設計議論 = GitHub Issues / PR コメント
  - 進捗一覧 = WMS Project Hub（GitHub Pages）
- **先生のアクション**：
  - 朝イチ判断シート（`specs/MORNING_DECISION_SHEET.md`）の「今井先生コメント」列に記入
  - PR 上で設計レビューコメント
  - 必要なら GitHub Issue 起票

### 2-3. 週次定例（30分）

- **頻度**：週1回・30分
- **議題**：
  - 致命傷ライン詰め（時吉さん×先生×まーちゃん）
  - 実装方針の承認
  - 翌週の優先順位
- **準備**：まーちゃんが事前に「議題 5 項目 + 各論点の要点」を提示。先生の時間を最大効率で使う。

## 3. レビュー範囲（先生に見てもらうもの）

### 必ず見てもらう

- **致命傷ライン15項目** の決定（朝イチ判断シート経由）
- **ER 図ドラフト**（specs/er_diagram_core.md）
- **マイグレーション SQL**（実 apply 前）
- **致命傷ラインに関わる主要 PR**（DB スキーマ変更・荷主切替・権限）

### 見せない（雑務）

- AI が起票した日常的な todos（量が多すぎ・全件は見せない）
- LINE Bot 設定・worker 改修などの **インフラ系**
- ドキュメント書き換え（CONTRIBUTING.md / MA_CHAN_BRIEFING.md 等）

> 例外：先生が「この種類は見たい」と希望されれば追加可能。

## 4. 緊急判断ルート

### レベル1：軽微な疑問（即時不要）

- GitHub Issue / PR コメント経由で記述
- 先生は次の定例 or 週内に回答

### レベル2：実装ブロッカー（24時間以内に欲しい）

- まーちゃんが時吉さんに依頼
- 時吉さんが LINE で先生に問い合わせ
- 先生回答を時吉さんが PM（まーちゃん）にフィードバック

### レベル3：致命傷ライン覆る（即対応）

- 時吉さんが LINE で先生に至急連絡
- 必要なら緊急ミーティング設定
- 該当工程の作業を **一時停止**して仕様再判断

---

## 5. 朝イチ判断シートでの活用

`specs/MORNING_DECISION_SHEET.md` に「今井先生コメント」列を追加済み。

### 流れ

1. **PM（まーちゃん）が朝イチ判断シートを更新**（推奨案・選択肢を整理）
2. **時吉さんが先生に「これ見てコメントくれ」と LINE で連絡**
3. **先生が GitHub 上で「今井先生コメント」列に記入**（または LINE で時吉さんに伝える→時吉さんが記入）
4. **時吉さんが「時吉さん判断」列に最終決定を記入**
5. **PM が確定した判断を specs に反映＋実装着手**

### 例：DB-1 在庫の持ち方

```
| DB-1 | 在庫の持ち方 | A/B/C/D 選択肢 | まーちゃん推奨: D | 今井先生コメント: 「Dでよいが location_id が NULL を許容する SKU マスタ的な持ち方も検討余地あり」 | 時吉さん判断: D（先生コメント踏まえて運用フェーズで再評価） |
```

## 6. 先生に時吉さんから提案・確認すべき項目（PM まーちゃんからのお願い）

> 時吉さん経由で先生に伝えてください。

### 6-1. 役割合意（最重要）

- 「テクニカルアドバイザー兼設計ゲートキーパー」（A+B+D 寄り・C 実装はやらない）でよいか
- 関与時間 週 5〜10 時間で問題ないか

### 6-2. レビュー範囲合意

- 上記「3. レビュー範囲」のとおりでよいか
- 追加で見たいもの・除外したいものはあるか

### 6-3. 連絡頻度・タイミング

- 週次定例の希望曜日・時間帯
- 緊急連絡 OK な時間帯（先生のお仕事との兼ね合い）

### 6-4. 業務委託・守秘義務契約

- 契約書の有無
- 報酬体系（時間給 / 月額 / 成功報酬）
- 守秘義務範囲（顧客情報・技術情報）

### 6-5. 最初のレビュー対象

- 朝イチ判断シート（`specs/MORNING_DECISION_SHEET.md`）の致命傷ライン15項目
- これに先生コメントをいただければ、Phase 8 実装フェーズ着手の道が開く

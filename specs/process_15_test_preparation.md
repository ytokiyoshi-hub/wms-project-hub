## Q1：3層テスト体系と役割分担

| 項目 | 内容 |
|------|------|
| 論点 | unit / E2E / 業務シナリオ の3層テストを、致命傷ライン15項目にどう対応させるか |
| 選択肢A | **Unit 最優先**：DB 関数・引当ロジック・RLS を unit で固める。E2E・業務シナリオは後回し |
| 選択肢B | **E2E 最優先**：業務フロー全体を Playwright で流す。unit は省略 |
| 選択肢C | **業務シナリオ最優先**：auto-verifier.js を軸に実シナリオ中心。unit・E2E は任意 |
| 選択肢D | **致命傷ライン二重確認**：致命傷ライン15項目 = unit（論理）+ E2E（動作）の二重確認。許容ライン（UI・個別機能）= 業務シナリオのみ |
| まーちゃん推奨 | **D** |
| 推奨理由 | RLS・引当・請求計算は「論理的に正しいか（unit）」と「実際の画面操作で壊れないか（E2E）」の両方を確認しないと安全とは言えない。UIや個別機能は業務シナリオで9割カバーすれば充分（時吉さん哲学の致命傷/許容の分離を踏襲）|
| 時吉さん回答 | （未回答）|

### D案 詳細：3層の責任境界

| 層 | 対象 | 主担当 | ツール候補 |
|----|------|--------|-----------|
| **Unit** | 引当ロジック・RLS ポリシー・計算式関数 | こーちゃん（実装時に同時作成）| Vitest（Next.js 環境）+ Supabase local |
| **E2E** | 業務フロー（入荷→検品→入庫→ピッキング→出荷）・荷主切替動作 | にーちゃん 主導 | Playwright |
| **業務シナリオ** | 実際の3PL業務フロー一気通貫・帳票確認 | にーちゃん 主導 | auto-verifier.js 拡張（WMS_SCENARIOS 追加）|

## Q3：テスト環境設計

| 項目 | 内容 |
|------|------|
| 論点 | テスト実行環境をどう設計するか（本番データ汚染ゼロ・CI 自動実行・失敗パターン4/8 防止）|
| 選択肢A | **staging のみ**：Vercel Preview URL で全テスト実行。Supabase は本番共用 |
| 選択肢B | **Supabase Branches**：テスト用 DB を本番と分離。schema は同じ、データは seed のみ |
| 選択肢C | **GitHub Actions**：PR ごとに自動実行。テスト DB は Supabase local（CI 上で起動）|
| 選択肢D | **B + C 組み合わせ**：Supabase Branch（テスト用 DB 分離）+ GitHub Actions（PR 時自動実行）|
| まーちゃん推奨 | **D** |
| 推奨理由 | Supabase Branches でテスト DB を本番と完全分離することで誤データ混入リスクをゼロにする。GitHub Actions + PR hook で「検証なし merge」を構造的に防止（失敗パターン4・8 の根絶。時吉さんが「staging 見てください」と言わなくても CI が自動で弾く体制）|
| 時吉さん回答 | （未回答）|

### D案 詳細：環境構成

```
[開発者（こーちゃん）]
  ↓ PR 作成
[GitHub Actions]
  ↓ Supabase Branch 自動生成（PR ごとの isolated DB）
  ↓ seed データ投入
  ↓ unit テスト実行（Vitest）
  ↓ E2E テスト実行（Playwright × にーちゃん管理）
  ↓ DB シナリオ実行（auto-verifier.js --db-verify）
  ↓ 全テスト green → merge 可能
  ↓ 全テスト red → にーちゃん ma_chan_outbox で通知
[Supabase Branch 自動削除（merge 後）]
```

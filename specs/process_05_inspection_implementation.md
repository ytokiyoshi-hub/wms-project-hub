### F-503：検品履歴・トレース機能

| 項目 | 内容 |
|------|------|
| 機能 | ロット番号・シリアル番号から検品履歴を逆引き |
| 用途 | リコール対応・荷主からの問い合わせ対応 |
| 出力 | 検索画面（PC） |
| 致命傷ライン | DB-2 / DB-3 |
| 9割突っ走り部分 | 検索 UI |
| 完璧に詰める部分 | クエリ性能（ロット×SKU の複合インデックス） |

**主要 SELECT：**
```sql
-- ロット番号からの完全トレース
SELECT
  l.lot_number, l.mfg_date, l.expiry_date,
  s.sku_code, s.jan,
  i.quantity, i.status, i.location_id,
  ir.inspection_type, ir.result, ir.inspected_at, ir.inspected_by
FROM lots l
JOIN inventory i ON i.lot_id = l.id
JOIN skus s ON s.id = l.sku_id
LEFT JOIN inspection_results ir ON ir.lot_id = l.id
WHERE l.lot_number = ?;
```

## QA確定事項（3号ヒアリング・QA-10）

### 入荷差異の承認アクション（確定）

入荷差異（数量差・品違い・破損）が発生した際の承認アクションは、**権限マスター（機能×ロール×可否マトリクス）** で切替制とする。

| 承認アクション | 内容 |
|--------------|------|
| 実数受入 | 差異がある数量のまま受入を確定する |
| 再検品要求 | 差異内容を保留し、再検品を指示する |
| 仕入先クレーム | 仕入先への差異通知・クレーム起票を行う |

#### 権限マスターの設計方針

```
permission_matrix（機能×ロール×可否）
  feature_code  = 'inbound_discrepancy_approve'
  role          = 'operator' / 'supervisor' / 'admin'
  allowed_action = 'receive_as_is' / 'request_reinspection' / 'file_claim'
```

- **前提**：B-7 ユーザー・ロールマスタ確定が必要
- 承認アクションのデフォルト（権限未設定時）は「再検品要求」
- ロール別に許可するアクションをマトリクスで管理し、UI 側は許可済みアクションのみ表示する

> F-504（検品 NG 時のフロー）とも連携。差異発生 → 権限確認 → アクション選択 → 記録 の順で処理する。

### 検品差異種別（discrepancy_type）

検品時に検出された差異の種別。`inbound_discrepancies.discrepancy_type` カラムで管理する enum 値。

| 値 | 表示名 | 説明 |
|---|---|---|
| `quantity` | 数量差異 | 予定数と実数の数量相違。`difference` カラムの符号で過剰／不足を表現 |
| `quality` | 品質不良 | 破損・期限切れ・隔離対象等の品質問題 |
| `wrong_item` | 誤納品 | 別商品の納品（商品コード不一致等） |

詳細補足は `reason_detail TEXT` に自由記述で記録する（例：「外装つぶれ」「賞味期限14日切れ」「同シリーズ別SKU」）。

#### 設計方針

- 「数量過剰／数量不足」は分離せず `quantity` に統合し、`difference` の符号で表現（モック `inspection-discrepancy-PC.html` 準拠）
- 「破損／期限切れ／隔離」は品質問題として `quality` に集約。区別が必要な場合は `reason_detail` で補足
- 上記3値域は画面モック `inspection-discrepancy-PC.html`（2026-05-17 最新）のフィルタ・サマリの差異区分に準拠
- 承認アクション（実数受入／再検品要求／仕入先クレーム）は本 enum と独立し、`permission_matrix` で切替

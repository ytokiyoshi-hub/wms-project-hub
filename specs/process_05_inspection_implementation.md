# 工程5：検品機能開発（マスタ管理＋トレース）

> ⚠️ 工程4（入荷・受入）の F-403 検品実行と一部重複。本工程は **検品マスタ管理＋トレース＋出荷検品** にフォーカス。

---

## スコープ

- 入荷検品（F-403）の **マスタ管理**（検品ルール・抜き取りパターン）
- 出荷検品（出荷直前の最終確認）
- 検品履歴・トレース機能（ロット追跡）

## 機能リスト

### F-501：検品ルールマスタ管理

| 項目 | 内容 |
|------|------|
| 機能 | 荷主×SKU 単位の検品ルールを管理 |
| 入力経路 | 管理画面（PC） |
| 出力 | `inspection_rules` テーブル |
| 致命傷ライン | BF-1 / DB-3 |
| 9割突っ走り部分 | 管理画面 UI |
| 完璧に詰める部分 | 抜き取り率・サンプル数の決定ロジック |

**主要カラム：**
```
inspection_rules
- id BIGINT PK
- owner_id BIGINT FK
- sku_id BIGINT FK NULL  -- NULL は荷主全体のデフォルトルール
- inspection_type TEXT   -- full / sampling / sign / serial_only
- sample_count INT NULL
- sample_percent NUMERIC NULL
- valid_from DATE
- valid_to DATE
```

### F-502：出荷検品（最終確認）

| 項目 | 内容 |
|------|------|
| 機能 | ピッキング完了後、出荷直前の検品 |
| 入力経路 | HT（バーコード読取） |
| 出力 | `outbound_inspections` テーブル |
| 致命傷ライン | BF-4（誤出荷防止） |
| 9割突っ走り部分 | 完了演出 |
| 完璧に詰める部分 | 誤出荷検知時のリカバリーフロー（在庫戻し） |

**ロジック：**
- 出荷指示の SKU/数量と HT 読取結果を突合
- 不一致なら NG → ピッカーに再ピックを指示 or 別オペレーターを呼ぶ
- OK なら `outbound_movements.status = 'inspected'` に更新

### F-503：検品履歴・トレース機能

| 項目 | 内容 |
|------|------|
| 機能 | ロット番号・シリアル番号から検品履歴を逆引き |
| 用途 | リコール対応・荷主からの問い合わせ対応 |
| 出力 | 検索画面（PC） |
| 致命傷ライン | DB-2 / DB-3 |
| 9割突っ走り部分 | 検索 UI |
| 完璧に詰める部分 | クエリ性能（ロット×SKU の複合インデックス） |

**主要 SELECT：**
```sql
-- ロット番号からの完全トレース
SELECT
  l.lot_number, l.mfg_date, l.expiry_date,
  s.sku_code, s.jan,
  i.quantity, i.status, i.location_id,
  ir.inspection_type, ir.result, ir.inspected_at, ir.inspected_by
FROM lots l
JOIN inventory i ON i.lot_id = l.id
JOIN skus s ON s.id = l.sku_id
LEFT JOIN inspection_results ir ON ir.lot_id = l.id
WHERE l.lot_number = ?;
```

### F-504：検品 NG 時のフロー

| 項目 | 内容 |
|------|------|
| 機能 | 検品で NG 判定された商品の取り扱い |

| 出力 | `inspection_ng_records` テーブル / 荷主通知 |
| 致命傷ライン | BF-4 |
| 9割突っ走り部分 | 通知文面 |
| 完璧に詰める部分 | NG 商品の `inventory.status` 遷移ルール |

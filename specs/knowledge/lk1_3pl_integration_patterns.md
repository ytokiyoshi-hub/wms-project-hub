# LK-1 深掘り：3PL 業界別 外部連携パターン

> Phase 9-O1（こーちゃん）/ 2026-05-09
> LK-1（外部連携方式）を「D=全方式+荷主切替」とする根拠を、業界別連携パターンで裏付ける。
> 対象：卸システム（BtoB）・ECモール・WMS競合 の3セグメント。

---

## 1. なぜ「全方式対応」が必要か

3PL が同一倉庫で複数荷主を受託すると、荷主ごとに連携方式が異なる現実がある。

| 荷主タイプ | 代表的な連携方式 | 理由 |
|-----------|----------------|------|
| 食品卸（大手流通向け）| EDI（流通BMS / JCA手順）| スーパー・コンビニが指定 |
| EC専業荷主（楽天・Amazon）| REST API（OMS経由）| 発送完了通知が必須 |
| 中小荷主・地場卸 | CSV + メール / FTP | IT投資が限られる |
| 大手メーカー荷主 | ERP 連携（SAP/NetSuite）| 社内ERPが指令塔 |

一つの倉庫でこれら全タイプが混在する。  
「A社はEDIだけ」では荷主獲得に上限が生まれる。**D案（全方式 + 荷主切替）が必須**。

## 2. 卸システム（BtoB）の連携パターン

### 2-1. 流通BMS（Business Message Standards）

**対象荷主：** 大手スーパー・コンビニ・ドラッグストア向けに商品を供給する食品卸・メーカー

| 項目 | 内容 |
|------|------|
| 策定主体 | GS1 Japan（流通業界団体）|
| フォーマット | XML over ebXML / AS2 |
| 主な電文 | 受注（DESADV）/ 出荷指示（ORDERS）/ 請求（INVOIC）/ 支払（REMADV）|
| 採用企業例 | イオン、セブン&アイ、ユニー・ファミリーマート、ウエルシア等 |
| 3PL への影響 | 荷主（食品卸）が大手流通チェーンから EDI で受発注 → 3PL への出荷指示も流通BMS 経由になるケースあり |

**WMS 実装の観点：**  
- 流通BMS は「荷主の取引先チェーン側」が要求する仕様であり、WMS が直接対応するより荷主側の EDI 変換サーバーを通してもらう方が現実的  
- WMS 側は「出荷指示の受信（CSV or API）」を受け取れれば OK  
- ただし大手荷主の自動連携では「WMS が AS2 エンドポイントを持つ」要求が来うる → LK-1=D で将来対応の口を残す

### 2-2. JCA 手順 EDI

**対象荷主：** 旧来の流通 EDI 標準を使い続けている食品・日用品卸

| 項目 | 内容 |
|------|------|
| 策定主体 | 日本チェーンストア協会（JCA）|
| フォーマット | 固定長テキスト（改行区切り） / ORCA 手順 |
| プロトコル | 公衆回線 / VAN 経由 |
| 現状 | 老朽化が進むが大手取引先が要求するため使い続けるケース多数 |

**WMS への示唆：** 直接対応より「EDI 変換 SaaS（PCA-EDI / ValueClick EDI）」をハブにした間接連携を推奨。WMS は変換後の CSV / API を受け取る設計で十分。

### 2-3. CSV / FTP 型（最多数の中小荷主）

中小食品卸・地場メーカーが最も多く使うパターン。

| 用途 | フォーマット | 運用 |
|------|------------|------|
| 入荷予定データ | CSV（荷主独自） | メール添付 or FTP アップロード |
| 出荷指示 | CSV（荷主独自） | 同上 |
| 在庫照会結果 | CSV / PDF | WMS から自動送信 |
| 出荷実績 | CSV | WMS から FTP 送信 or ポータル表示 |

**設計ポイント：**
- 荷主ごとに CSV フォーマットが異なる（項目順・ヘッダ有無・文字コード）
- `csv_format_definitions` テーブル（荷主 × 種別 × 項目マッピング）で吸収する設計が必須
- 文字コード：UTF-8 と Shift-JIS の両対応（食品卸では Shift-JIS が現存）

### 2-4. ERP 連携（大手荷主）

| ERP | 連携方式 | 実装負荷 |
|-----|---------|---------|
| SAP S/4HANA | RFC / IDOCs / REST API | 重い（要 SAP コンサル） |
| Oracle NetSuite | REST API（SuiteQL）| 中程度 |
| Microsoft Dynamics 365 | Dataverse API | 中程度 |
| 弥生・freee（国内中小）| CSV 出力 / API | 軽い |

**3PL への示唆：** 大手メーカーを荷主として獲得すると SAP 連携を要求されることがある。これを「連携費用無料」の範囲に含めるか、別途相談とするかは CA-1（請求賃率）の設計に影響する。

## 3. EC モール連携パターン

EC 物流では「受注 → 出荷指示 → 出荷 → 発送完了通知」のサイクルが基本。

### 3-1. 主要ECモール別 API 特性

| モール | API / 連携方式 | 発送完了通知の必須性 | 備考 |
|--------|--------------|-------------------|------|
| **楽天市場** | RMS API（受注取得・在庫更新）/ CSV | ✅ 必須（遅延でペナルティ）| 送料無料ライン管理も連携 |
| **Amazon** | SP-API（旧 MWS）/ FBA | ✅ 必須（追跡番号登録）| FBA 使用時は WMS 不使用 |
| **Yahoo!ショッピング** | Yahoo! 注文 API / CSV | ✅ 必須 | ストアクリエイターPro との並列管理 |
| **ZOZO / ZOZOTOWN** | ZOZO 独自 API | ✅ 必須 | ハンガー・バーコード独自仕様あり |
| **Shopify** | Admin API（GraphQL/REST）| ✅ 必須 | 最も標準化・実装しやすい |
| **Qoo10** | Qoo10 API | ✅ 必須 | 海外向け商品が多い |
| **au PAY マーケット** | 受注 API | ✅ 必須 | KDDI 系 |

### 3-2. OMS（受注管理システム）ハブ型アーキテクチャ

**EC 荷主が複数モールに出店している場合、WMS が各モールの API を直接叩くのは非効率**。  
現実的な実装は「OMS を中継ハブにする」設計。

```
[ 楽天 RMS ]  ─┐
[ Amazon SP-API] ─┤
[ Yahoo! API ]  ─┤ → [ OMS（ネクストエンジン等）] → [ WMS（自社）] → 出荷
[ Shopify API ] ─┘              ↑ 出荷完了通知を返す
```

**主要 OMS（国内市場シェア）：**

| OMS | 特徴 | 連携モール数 |
|-----|------|------------|
| **ネクストエンジン**（Hamee）| 国内最大シェア。楽天・Amazon・Yahoo!・ZOZO 等を網羅 | 70+ |
| **CROSS MALL**（アイルコーポレーション）| 大手流通にも強い。API 連携が充実 | 50+ |
| **GoQSystem** | 中小 EC 向け。安価 | 30+ |
| **LOGILESS** | WMS+OMS 一体型（競合）| 自社で閉じている |
| **TEMPOSTAR**（ソフテル）| 受注管理特化 | 50+ |

**WMS 実装の示唆：**  
- **ネクストエンジン連携 API の実装が最優先**（国内 EC 荷主の大半をカバーできる）
- OMS 連携 = 出荷指示を受け取る + 出荷完了通知を返す の2方向
- 自社 WMS が「ネクストエンジン対応 WMS」として登録されることが EC 荷主獲得の近道

### 3-3. 発送完了通知の仕様

EC 物流で最も重要な「返信」処理。

| 必須情報 | 内容 |
|---------|------|
| 注文 ID | モールの受注番号（照合キー）|
| 配送業者コード | ヤマト（0001）/ 佐川（0002）/ 日本郵便（0003）等 |
| 送り状番号（追跡番号）| 配送業者が発行する番号 |
| 発送日 | YYYY-MM-DD |

**実装の落とし穴：**  
- 発送完了通知の遅延は「出荷停止」処分の原因になる（楽天の場合特に厳格）
- 出荷後 15 分以内の通知が理想。バッチ（日 1 回）は不可
- WMS が送り状番号を記録してすぐ通知するフロー（F-1402 Webhook の強化）が必要

### 3-4. EC 物流特有の WMS 要件（連携から逆算）

| 要件 | 理由 |
|------|------|
| 受注番号を入荷・出荷レコードに保持 | 発送完了通知で照合キーになる |
| 荷主別 OMS 接続設定（connectors テーブル）| 荷主A=ネクストエンジン、荷主B=Shopify など |
| 出荷完了イベント → 即時 Webhook 発火 | 発送完了通知の即時性 |
| 配送業者コード・送り状番号の記録 | 通知に必須 |

## 4. WMS 競合の連携アーキテクチャ

### 4-1. ロジザードZERO

| 項目 | 内容 |
|------|------|
| 連携方式 | REST API + CSV/FTP + 独自連携アダプター |
| 標準連携先 | 楽天・Amazon・Yahoo!・ZOZO・ネクストエンジン・CROSS MALL・GoQSystem・Shopify 等 |
| **連携課金** | **追加あり**（連携数・月間件数に応じた従量オプション）|
| EDI 対応 | あり（オプション扱い・追加費用）|
| 弱点 | 価格非公開・高額。中小荷主への連携費用が導入障壁になるケースあり |

**自社 WMS の差別化：**  
ロジザードが「連携費用追加」にしている部分を「**連携費用無料**」とすることで中小3PL にアピールできる。

### 4-2. ロジクラ

| 項目 | 内容 |
|------|------|
| 連携方式 | Shopify API 特化 + ネクストエンジン |
| 強み | iPhone + Shopify のセットアップが圧倒的に簡単 |
| 弱点 | 卸（EDI）・大手ERP 連携が弱い。iOS 専用（Android 非対応）|

**自社 WMS の差別化：**  
キーエンス HT（LK-2）+ EDI 対応で卸荷主も取れる点が差別化。

### 4-3. LOGILESS（最注意競合）

| 項目 | 内容 |
|------|------|
| 連携方式 | WMS + OMS 一体型 |
| 強み | EC 荷主は OMS 費用が不要・楽天・Amazon・Yahoo!・Shopify を自社内で完結 |
| 弱点 | 卸（BtoB）・複数荷主 3PL 運営には不向き。マルチテナント機能が弱い |

**自社 WMS の差別化：**  
LOGILESS は「EC に特化した荷主1社向け倉庫管理」に近い。**複数荷主の 3PL 運営**では自社 WMS の荷主切替機能・マルチテナント RLS が圧倒的に強い。

### 4-4. ZAICO / TEMPOSTAR（参考）

- ZAICO: 在庫管理特化（クラウド棚卸）。WMS というより在庫 SaaS。競合性は低い
- TEMPOSTAR: 受注管理 OMS。WMS 機能なし → ネクストエンジン連携先として共存する

## 5. LK-1=D 推奨の根拠サマリー

| 荷主パターン | 必要な連携方式 | D 案で対応 |
|------------|-------------|----------|
| 食品卸（大手流通向け）| 流通BMS / JCA 手順（EDI 変換後CSV）| ✅ CSV + 将来 EDI |
| EC 専業（楽天・Amazon 複数モール）| OMS（ネクストエンジン）経由 API | ✅ API |
| 中小荷主（地場卸・小規模 EC）| CSV + メール | ✅ CSV |
| 大手メーカー荷主 | SAP / NetSuite API + EDI | ✅ API + EDI（将来）|

A（CSV のみ）・B（API のみ）・C（EDI のみ）では**同一倉庫で複数荷主が混在するマルチテナント 3PL には対応不能**。  
D が唯一の選択肢。

## 6. 実装優先順位（LK-1 = D が確定した場合）

Phase 9 Stage 4 で LK-1 を実装する際の優先順位案：

| 優先 | 連携方式 | 理由 | 想定工数 |
|------|---------|------|---------|
| **最優先** | ネクストエンジン API 連携（受信 + 完了通知）| EC 荷主の大多数をカバー | 3 日 |
| **次優先** | CSV インポート / エクスポート（荷主別フォーマット定義）| 卸荷主・中小荷主をカバー | 2 日 |
| **次次優先** | Shopify Webhook 受信（出荷完了通知）| D2C 荷主の増加対応 | 1 日 |
| **将来対応** | EDI（JCA 手順 / 流通BMS 経由）| 大手流通向け荷主獲得時に追加 | 5+ 日 |

合計（優先3項目）: 6 日 → Phase 9 の 6 人日見積とほぼ一致（`PHASE9_IMPLEMENTATION_PLAN.md` Stage 4 LK-1=6日）。

## 7. 自社 WMS の差別化訴求ポイント（営業資料向け）

連携の観点から以下を訴求できる：

```
【3PL 向け WMS が連携で差別化できる点】

① 荷主を選ばない：EDI / API / CSV すべて対応
   → 食品卸荷主も EC 荷主も同じ WMS で受託できる

② 連携費用ゼロ：荷主が増えても連携追加料金なし
   → ロジザードは連携追加で課金。自社は設定のみで対応

③ ネクストエンジン対応 WMS として登録可能
   → EC 荷主は「ネクストエンジン対応か」を導入基準に使う

④ 発送完了通知を即時発火
   → 出荷後 15 分以内に楽天・Amazon に通知可能（ペナルティゼロ）

⑤ 荷主ごとに連携設定を分離（マルチテナント）
   → 荷主A の API 設定が荷主B に漏れない・影響しない（RLS 設計）
```

## 8. 参考情報

### 流通BMS
- 策定：GS1 Japan（https://www.gs1jp.org）
- 電文仕様：DESADV / ORDERS / INVOIC / REMADV

### ネクストエンジン WMS 連携仕様
- API ドキュメント（アライアンス申請後に取得）
- 「WMS 連携パートナー登録」制度あり → 登録で EC 荷主への紹介経路になる

### Amazon SP-API
- 公式ドキュメント：https://developer-docs.amazon.com
- Fulfillment Outbound API でレポート取得 / 発送確認登録

### Shopify Fulfillment API
- 公式ドキュメント：https://shopify.dev/docs/api/admin-rest/fulfillments
- 3PL 向け：Fulfillment Service API（カスタム倉庫登録可能）

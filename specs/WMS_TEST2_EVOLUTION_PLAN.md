## 1. 背景

### きっかけ
2026-05-28 時吉さんが `localhost:8778`（= wms-test2 ローカル）を見て「**今のままでは使えない**」「**新しいURLに使用感がわかるものが欲しい**」と判断。

### 現状分析（壁打ち時点）
3リポジトリが存在するが、**統合された「本番品質で触れる1つのアプリ」がない**：

| リポ | 役割 | 状態 | 公開 |
|---|---|---|---|
| `wms-project-hub` | 仕様・モック (1号管理) | 67 HTMLファイル | ✅ GitHub Pages 公開 |
| `wms-test2` | UI実装集 (時吉さん主導) | 472画面・**421画面に統一構造適用済** | ローカル localhost:8778 のみ |
| `wms-impl` | 動作プロトタイプ (2号管理) | 25工程/45画面 + SQLite | ローカル localhost:8777 のみ |

→ 「画面はあるが業務として回らない / データが繋がってない / 使用感が薄い」

### 本計画の本旨
- **wms-test2 をマスター**にして、本番品質の統合アプリへ進化させる
- 公開URLで時吉さんが触れる状態を作り、「使用感」を担保する
- 中津さんの本番DB/APIに接続し、最終的に実倉庫運用可能なレベルへ

## 5. 技術スタック方針（2026-05-28 修正版）

> **⚠️ 重要修正**: wms-test2 詳細調査の結果、既に Phase 9 Supabase 移行コードが完備されていることが判明（migrations/001-003 + adapters/supabase-adapter.js 雛形）。Next.js 完全書き換えは 4-5週かかるため MVP 1-2週要望と矛盾。
> **修正方針**: MVPは「**Express維持＋Supabase移行＋Vercel公開**」。フロント React化は後段で段階移行。

### MVPフェーズ（1-2週）
- **フロント**: 既存 wms-test2 の純HTML/CSS/JS をそのまま維持（**482画面**）
- **サーバ**: 既存 `server/index.js` (Express 4.19.2 / 2,668行 / 164エンドポイント) を維持
- **DB**: SQLite → **Supabase PostgreSQL**（`migrations/001-003` 適用済 SQL 投入）
- **認証**: **Supabase Auth**（`adapters/supabase-adapter.js` 既存雛形を完成）
- **RLS**: **Supabase RLS** 有効化（`migrations/002_rls_policies.sql` を投入）
- **ホスティング**: **Vercel**（Express を Vercel Functions or Edge で動かす構成検討）
- **検証**: 既存 19シナリオ + 1年BULK 21k件 を Supabase 環境で再走→ PASS で本番品質

### ベータフェーズ以降
- フロント部分的に React化（領域別に段階移行）
- 中津さん本番DB接続（フロント用 Supabase → 本番 Supabase へ統合 or 接続）
- 最終的な Next.js 移行可否は別途判断

### wms-test2 の既存資産（Explore 調査で判明）
| 資産 | 状態 |
|---|---|
| 画面数 | 482ファイル（PC434 + HT37 + app5 + 他） |
| API | 164エンドポイント（Express） |
| データモデル | 45テーブル + インデックス + FK |
| migrations | 001-003 で Postgres + RLS 完備 |
| Supabase adapter | 雛形あり（未稼働） |
| シナリオ | 43本（19本 PASS済み） |
| 負荷検証 | 21k件SO + 6.3k件入荷 / SLA 1秒以内 |
| HT実機対応 | キーエンス BT-A2000 `?real=1` mode |

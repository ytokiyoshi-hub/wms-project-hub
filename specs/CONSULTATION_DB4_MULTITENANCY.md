# 【中津さんご相談】WMS マルチテナント設計の選択

> 作成：2026-05-09 / 時吉（社長）
> 目的：WMS開発の最重要設計判断について中津さんのご意見をいただきたい

---

## 背景（1分で理解できる前提）

- 開発中のシステム：**WMS（倉庫管理システム）**
- 対象：中小3PL（物流アウトソーシング会社）向けSaaS
- 特徴：**1つの倉庫が複数の荷主（預け主）の荷物を管理する**

### 「荷主」とは？
倉庫に荷物を預ける会社のこと。A社・B社・C社それぞれが同じ倉庫を使う。

```
倉庫（ユーザー）
├── A社の荷物 → A社の担当者だけ見える
├── B社の荷物 → B社の担当者だけ見える
└── C社の荷物 → C社の担当者だけ見える
```

### 想定規模
- 荷主数：多くて50社程度（増減を繰り返す。撤退してもデータは残す）
- 1倉庫あたりの荷主：平均5〜20社程度

---

## 相談したい設計上の選択

**「荷主ごとのデータをどう分離するか」**

この選択は後から変えると開発工数が+10〜15人日発生する「最重要の設計判断」です。

---

## 選択肢の比較

### A案：物理分離（荷主ごとに独立したデータベース）

```
荷主Aの倉庫 → DBサーバーA（A社のデータのみ）
荷主Bの倉庫 → DBサーバーB（B社のデータのみ）
荷主Cの倉庫 → DBサーバーC（C社のデータのみ）
```

**メリット**
- データ漏洩リスクがゼロ（物理的に分離されている）
- 荷主ごとにDB設定を変えられる（将来的な柔軟性）
- 1社のDBが重くなっても他社に影響しない
- 荷主への説明・信頼獲得がしやすい（「あなた専用のDBです」）

**デメリット**
- **コスト：荷主1社につき約¥3,700/月（Supabase Pro）**
  - 50社 → **¥185,000/月**（年間¥220万）
  - 荷主が増えるほど運用コストが線形に増加
- スキーマ変更（テーブルの列追加など）のたびに50社分の作業が発生
- 新規荷主の追加に30分〜1時間の初期設定作業が必要

---

### B案：論理分離＋RLS（1つのDBに全荷主のデータを格納、アクセス制御で分離）

```
1つのDBサーバー
├── 全データ（owner_id列で荷主を識別）
│   ├── owner_id=A → A社のデータ
│   ├── owner_id=B → B社のデータ
│   └── owner_id=C → C社のデータ
└── RLS（行レベルセキュリティ）がアクセスを自動制限
    → ログインユーザーに紐づく荷主のデータしか取得できない
```

**RLS（Row Level Security）とは？**
PostgreSQL（Supabase）の機能。「このユーザーはこのowner_idの行しか読み書きできない」をDB側で強制する仕組み。アプリ側のコードミスがあっても、DB側で防ぐ。

**メリット**
- **コスト：荷主数に関わらず¥3,700〜¥15,000/月**（データ量次第。50社でも変わらない）
- スキーマ変更は1回で全荷主に適用
- 新規荷主の追加はowner_idの発行のみ（1分以内）

**デメリット**
- **RLSの実装にバグがあると、A社のデータがB社に見える可能性がある**
- RLSポリシーのテストが複雑（目に見えない制御のため）
- 1つのDBに全データが集まるため、大量データ時のパフォーマンス設計が必要

---

## コスト比較（月額・荷主50社想定）

| | A案（物理分離） | B案（論理分離+RLS） |
|--|:---:|:---:|
| 月額インフラコスト | **¥185,000〜** | **¥3,700〜¥15,000** |
| 荷主増加時のコスト | 1社追加 ≒ +¥3,700/月 | ほぼ変わらない |
| スキーマ変更コスト | 荷主数×作業時間 | 1回 |
| 初期設定（荷主追加） | 30分〜1時間 | 1分以内 |

---

## 開発への影響

この判断は**全テーブルの設計・全APIの設計・認証設計・請求計算設計**に影響します。

| 影響箇所 | A案 | B案 |
|----------|-----|-----|
| テーブル設計 | owner_id不要 | 全テーブルにowner_id必須 |
| API設計 | DB接続先を動的切替 | RLSポリシーで自動制御 |
| 認証設計 | 荷主ごとのプロジェクト | 共通プロジェクト内でロール管理 |
| 請求計算 | 荷主ごとのDB分離で自然に計算 | owner_idで集計 |
| 開発工数 | Phase9全体+10〜15人日 | 現行計画のまま（57人日） |

---

## 現在の開発チームの見解

**B案（論理分離+RLS）を推奨**

理由：
1. コスト差が年間約¥200万（SaaS事業としてスケールしにくい）
2. Supabaseの標準機能としてRLSが充実しており、実績豊富
3. 開発工数・スピード・運用コストすべてでB案が有利

ただし、以下の懸念を払拭できていない：
- RLSのセキュリティ品質をどう担保するか
- 荷主に対してデータ分離をどう説明・保証するか

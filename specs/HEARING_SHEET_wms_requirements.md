## ⚠️ WMS 運用前提（全設問共通・3号ヒアリング確定 2026-05-16）

| 項目 | 前提 |
|------|------|
| WMS の所有主体 | WMS は **倉庫の所有物**。荷主が直接操作するシステムではない |
| 荷主の操作範囲 | 荷主は在庫照会・発注登録・請求明細確認のみ。在庫移動・マスタ変更・フロー実行は倉庫管理者が担当 |

| マーク | 意味 |
|--------|------|

**Phase9実装依存度の基準**

| レベル | 意味 |
|--------|------|
| **高** | この回答がないと該当工程の実装着手ができない |
| **中** | 未回答のまま進むと後から設計変更が必要になるリスクがある |
| **低** | 実装後でも比較的容易に変更・追加が可能 |

## 工程10：返品処理（顧客返品）

> スコープ：出荷後に顧客から返品が届いた場合の受入→検品→在庫戻し/廃棄判定→記録のフロー  
> 出典：`specs/process_10_returns.md` Q1-Q8

### Q10-1：返品の原因分類方針 🏢

| 項目 | 内容 |
|------|------|
| 論点 | 「顧客都合返品」「倉庫ミスによる誤出荷返品」「品質問題による返品」を区別して管理するか |
| 選択肢A | 原因は区別しない（備考欄への記録のみ・全て同一フローで処理）|
| 選択肢B | 原因ごとに別フロー（費用負担者・在庫戻しルールが原因によって異なる）|
| 選択肢C | 荷主ごとに設定（原因区分が必要な荷主のみ有効化）|
| 仕様への影響 | `customer_returns.return_reason_type` カラム / 費用精算ロジック / 誤出荷費用の倉庫負担判定 |
| Phase9実装依存度 | **高**（CA-1 請求計算・費用精算に直結） |
| まーちゃん推奨 | **C**（荷主設定）：誤出荷の費用負担は荷主ごとに契約が違う。原因区分なしでは請求精算が詰まる可能性あり |
| 時吉さん回答 | **B（確定 2026-05-16）：全荷主で返品原因を区分管理（顧客都合／倉庫ミス／品質不良）。費用負担・在庫戻しルールが原因に連動。`customer_returns.return_reason_type` カラムに区分値を必須化。** |

### Q10-2：返品管理番号（RMA）の運用方法 🏢

| 項目 | 内容 |
|------|------|
| 論点 | 返品に対してWMSで管理番号（RMA番号）を発番・トラッキングするか |
| 選択肢A | WMSでは管理しない（荷主側で申請管理・WMSは物理受入から処理）|
| 選択肢B | WMSでRMA番号を発番・管理（物理受入前に事前登録が必要）|
| 選択肢C | 荷主システムからRMA番号を連携受取（WMSで発番せず受け取るのみ）|
| 選択肢D | 荷主ごとに切替（A/B/C を荷主ごとに選択可能）|
| 仕様への影響 | `customer_returns.rma_number` / `owners.return_strategy` への rma_required フラグ追加 |
| Phase9実装依存度 | **中**（受入フローの画面設計に影響）|
| まーちゃん推奨 | **D**（荷主別・最初はAから入って需要に応じてB/C追加）|
| 時吉さん回答 | **選択肢外（確定 2026-05-17）：返品入庫は通常入庫と区別せず入庫No.で一元管理する。RMA番号の運用有無は荷主マスターで切替（WMS発番／荷主連携受取／不使用の3方式から荷主ごとに選択）。** |

### Q10-3：返品品の状態区分（検品判定区分） 🏢

| 項目 | 内容 |
|------|------|
| 論点 | 返品品を検品したときの状態をどの粒度で分類するか |
| 選択肢A | 「良品」「不適合品」の2区分（シンプル・廃棄判定は別途）|
| 選択肢B | 「良品」「要確認」「廃棄確定」の3区分 |
| 選択肢C | 「良品」「要修理」「廃棄確定」「保留（荷主確認待ち）」の4区分 |
| 選択肢D | 荷主ごとに判定区分を設定可能（マスタで管理）|

| 時吉さん回答 | **選択肢外（確定 2026-05-16）：返品品専用の condition enum（return_inspections.condition）は新設しない。既存 `inventory.status` を返品品でも共用。返品入庫時の初期ステータスは荷主／ユーザーが `owners.return_strategy.return_default_status` で任意設定可（デフォルト「良品」固定なし）。** |

### Q10-4：良品の通常在庫への戻し承認ルール 🏢

| 項目 | 内容 |
|------|------|
| 論点 | 返品品が「良品」と判定された後、通常在庫に戻す際の承認ルール |
| 選択肢A | 検品完了後に即時自動で通常在庫に戻す（スタッフ判断で完結）|
| 選択肢B | 「返品良品」として一旦保留し、荷主確認または上長承認後に通常在庫に移動 |
| 選択肢C | 荷主ごとに選択（自動戻しOKな荷主はA・承認必須な荷主はB）|
| 仕様への影響 | `owners.return_strategy.auto_restock` フラグ / `inventory.status` 遷移フロー |
| Phase9実装依存度 | **高**（BF-4「在庫に載せず別ステータス」確定決定との整合性あり）|
| まーちゃん推奨 | **C**（荷主ごと設定）：食品・医薬品等の高品質管理荷主はB必須。一般品はAでOK |
| 時吉さん回答 | **選択肢C（確定 2026-05-17）：良品の通常在庫戻しは荷主ごとに切替（`owners.return_strategy.auto_restock` フラグで制御）。食品・医薬品等の高品質管理荷主は承認必須、一般品は自動戻しOK。** |

### Q10-5：廃棄確定の承認フロー 🏢

| 項目 | 内容 |
|------|------|
| 論点 | 返品品の廃棄を確定させる権限と、WMS内での承認記録の設計 |
| 選択肢A | WMS内承認フローなし（倉庫スタッフが判断・DB記録のみ）|
| 選択肢B | WMS内承認フロー（倉庫スタッフ起案 → 倉庫上長が承認）|
| 選択肢C | WMS内承認フロー（倉庫スタッフ起案 → 荷主の最終確認が必須）|
| 選択肢D | 荷主ごとに設定（高額品荷主はC・一般品はA）|
| 仕様への影響 | `disposal_approvals` テーブル（新設の可能性）/ AU-1 権限設計との整合 |
| Phase9実装依存度 | **高**（廃棄は取り消し不可のため権限設計が重要。後から変更困難）|
| まーちゃん推奨 | **D**（荷主別・基本はB）：廃棄は「記録だけ」は危険。最低でもBを基本にし高額品荷主はCを適用 |
| 時吉さん回答 | **C（確定 2026-05-16）：簿内品の廃棄は荷主の最終承認が必須（廃棄申請：倉庫スタッフ → 荷主の最終承認フロー）。倉庫上長のみのB案は寄託物の法的所有権が荷主にあるため不成立。簿外品はWMS管理外。** |

## 工程15：テスト準備（3層テスト体制）

> 社内チームでの意思決定が必要な事項。  
> 出典：`specs/process_15_test_preparation.md` Q1-Q3

### Q15-1：テスト体制と役割分担 📋

| 項目 | 内容 |
|------|------|
| 論点 | システムの品質担保のためのテスト体制をどう設計するか |
| 選択肢A | **単体テスト（Unit）最優先**：DB関数・引当ロジック・RLS権限を自動テストで固める。画面テストは後回し |
| 選択肢B | **E2Eテスト最優先**：業務フロー全体を画面操作テストで検証。単体テストは省略 |
| 選択肢C | **業務シナリオ最優先**：実際の3PL業務フロー一気通貫テストを中心に。単体/E2Eは任意 |
| 選択肢D | **致命傷ライン二重確認**：重要機能（引当・RLS・請求計算）は単体+E2Eの両方で検証。一般機能は業務シナリオのみ |
| 仕様への影響 | 開発工数配分 / 品質担保レベル / Stage別着手タイミング |
| Phase9実装依存度 | **高**（テスト設計が実装スケジュールに直結）|
| まーちゃん推奨 | **D**（致命傷ライン二重確認）|
| 時吉さん回答 | **D（確定 2026-05-16）：致命傷ライン（引当・RLS・請求計算）= unit（論理確認）+ E2E（動作確認）の二重確認。一般機能（UI・個別機能）= 業務シナリオのみ。加えて時吉さん監修「想定荷主モデルによる擬似業務テスト」を実施（模擬データ＋日々の運用データを数十件×1年分でバグ出し）。体制：こーちゃん=Unit / にーちゃん=E2E+業務シナリオ / 時吉さん=擬似テスト荷主モデル監修。擬似テストの期間・荷主モデル社数は未決。** |

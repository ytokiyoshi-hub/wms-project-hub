### F-1103：差異検知

| 項目 | 内容 |
|------|------|
| 機能 | 帳簿在庫 vs 実在庫を照合し差異を検知 |
| ロジック | `inventory_count_records.actual_qty - inventory.quantity` |
| 出力 | `inventory_count_diffs` テーブル |
| 致命傷ライン | DB-1 / CA-2（差異額） |
| 9割突っ走り部分 | 差異一覧画面 |
| 完璧に詰める部分 | 差異の深刻度判定（許容差異 0.1% 以下 vs 重大差異） |

**差異額計算（CA-2 移動平均ベース）：**
```sql
SELECT
  d.sku_id, d.lot_id, d.location_id,
  d.actual_qty - d.book_qty AS diff_qty,
  ABS(d.actual_qty - d.book_qty) * i.unit_cost AS diff_value
FROM inventory_count_diffs d
JOIN inventory i ON i.id = d.inventory_id
WHERE d.session_id = ?;
```

| 項目 | 内容 |
|------|------|
| 機能 | 確定した差異を在庫に反映する。**独立機能 `inventory_adjustments` として設計**（棚卸しセッションだけでなく日常的な在庫調整にも使用可能） |
| 承認フロー | **荷主承認フロー**：申請（倉庫スタッフ）→ 荷主承認 → 倉庫実行（AU-1 との整合） |
| 理由バリデーション | **調整理由コード（`adjustment_reason_code_id`）の選択を必須**とする。コードは `adjustment_reason_codes` マスタから選択（デフォルト：破損／紛失／数え誤り／盗難 等）。補足欄（`supplement_note` フリーテキスト）をオプションで併設。（QA-9 再確定 2026-05-17） |
| 出力 | `inventory_adjustments` テーブルに記録 / `inventory.quantity` 更新 / `inventory_movements` に履歴（type='adjustment'） |
| 致命傷ライン | DB-1 / AU-1 |
| 9割突っ走り部分 | 承認画面 UI |
| 完璧に詰める部分 | 監査証跡（誰が・いつ・何を補正したか）・荷主への報告書面 |

> **QA-9 再確定（2026-05-17）：** 在庫調整の理由入力は「調整理由コード（選択式）」を採用する。コードは `adjustment_reason_codes` マスタで管理し、デフォルト項目（破損／紛失／数え誤り／盗難 等）を事前登録。マスタ画面で追加・変更可能。コード選択に加えて補足欄（フリーテキスト）を併設。荷主承認と理由コード選択を必須とする。独立機能 `inventory_adjustments` として設計。
> ※ 経緯：3号ヒアリング（2026-05-16）では方式A（コードなし・備考欄のみ）で確定。実装フェーズ（2026-05-17）で時吉さんが方式B（選択式コード）に再確定。

```sql
-- 在庫調整テーブル（QA-9 確定設計）
CREATE TABLE inventory_adjustments (
  id                        BIGSERIAL PRIMARY KEY,
  inventory_count_diff_id   BIGINT REFERENCES inventory_count_diffs(id),  -- 棚卸し由来の場合
  inventory_id              BIGINT NOT NULL REFERENCES inventory(id),
  owner_id                  UUID NOT NULL REFERENCES owners(id),
  qty_before                NUMERIC NOT NULL,
  qty_after                 NUMERIC NOT NULL,
  adjustment_reason_code_id BIGINT NOT NULL REFERENCES adjustment_reason_codes(id),  -- 理由コード選択必須（QA-9 方式B）
  supplement_note           TEXT,                                                      -- 補足欄（フリーテキスト・任意）
  status                    TEXT NOT NULL DEFAULT 'pending'
                              CHECK (status IN ('pending', 'approved', 'rejected', 'executed')),
  requested_by              UUID NOT NULL REFERENCES users(id),  -- 申請者（倉庫スタッフ）
  approved_by               UUID REFERENCES users(id),           -- 承認者（荷主）
  executed_by               UUID REFERENCES users(id),           -- 実行者（倉庫スタッフ）
  requested_at              TIMESTAMPTZ DEFAULT NOW(),
  approved_at               TIMESTAMPTZ,
  executed_at               TIMESTAMPTZ,
  notes                     TEXT
);
```

## 棚卸方式の選択（Q11-1 確定・5種拡張・荷主単位切替）

> **Q11-1 確定（3号ヒアリング 2026-05-17）：** 棚卸方式を5種類に拡張。荷主ごとに方式を選択可能。同一荷主での複数モード併用にも対応する設計とする。各モードの凍結方針は QA-4 確定（F-1101/F-1101-B）の一斉/循環凍結動作を5種に展開し統合設計する。

| 方式 | 特徴 | 凍結方針 | 推奨ターゲット |
|------|------|---------|--------------|
| 日々棚卸 | 毎日少量ロケを担当者が確認 | **凍結なし** ─ 通常入出庫と並行運用可 | 高回転SKU・Aクラス商品 |
| ピッキング時在庫確認 | ピッキング作業と同時に在庫確認 | **凍結なし** ─ ピッキング完了後に確認値を記録 | 高回転倉庫・人員効率化 |
| 循環棚卸 | 365日均等にロケを数える | **凍結なし** ─ 通常入出庫と並行運用可（F-1101-B 設計準拠） | 規模中〜大・SKU多 |
| 月次棚卸 | 月1回・対象荷主の全在庫を計画的に確認 | **部分凍結** ─ 対象ロケを一時凍結・凍結中実行要求はエラー返却（F-1101 設計準拠） | 月次精度報告が必要な荷主 |
| 期末棚卸 | 年1回・全在庫を一度に数える・倉庫休止 | **全体凍結** ─ 対象ロケを凍結・凍結中実行要求はエラー返却（F-1101 設計準拠） | 規模小・SKU少／決算期対応 |

**荷主マスタでの設定：** `owners.inventory_modes` に5種のいずれか（または複数）を設定可能。複数モード併用時は各棚卸セッション起動時に方式を指定する。

## 在庫データ修正の取扱い（Q13-2 確定・在庫側）

> **Q13-2 確定（3号ヒアリング 2026-05-17）：** 在庫データの修正ルールを以下の通り確定する。請求側の取扱いは別タスク H1R-D が担当。

### 修正経路の制限

- **在庫データ修正は `inventory_adjustments` 経由のみ可能**（F-1105 設計準拠）
- 在庫テーブル（`inventory`）への直接 UPDATE は禁止。RLS ポリシーで倉庫スタッフ・荷主ロールから `inventory` への直接 UPDATE を拒否する
- 修正が必要な場合は必ず `inventory_adjustments` に申請レコードを作成し、承認フロー（申請 → 荷主承認 → 倉庫実行）を経て反映する

### 過去データの保護

- **過去データの直接書き換えは禁止**
- `inventory_movements`（在庫移動履歴）は immutable。作成後の UPDATE・DELETE は禁止（RLS + トリガーで保護）
- 誤記訂正が必要な場合は訂正レコードを新たに追加する（打ち消し＋正規）方式とする

### 月次スナップショットの確定値保護

- 月次スナップショット（`inventory_snapshots`）は月次締め処理完了後に `locked = true` へセットし、確定値として保持する
- `locked = true` のスナップショットは変更不可（RLS ポリシーで UPDATE を拒否）
- 月次締め後に差異が発覚した場合は、翌月の調整として `inventory_adjustments` を新規起票する

```sql
-- 月次スナップショット保護イメージ
-- locked=true のレコードへの UPDATE を RLS で拒否する
CREATE POLICY "snapshot_locked_no_update" ON inventory_snapshots
  FOR UPDATE
  USING (locked = false);  -- locked=true のレコードは更新対象外
```

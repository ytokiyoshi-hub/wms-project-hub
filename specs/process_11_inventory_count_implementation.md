# 工程11：棚卸機能開発（在庫精度の保証）

> 3PL の信頼の源＝在庫精度（差異 0.1% 以下）。誤差は荷主との関係性に直結。
> 致命傷ライン BF-4（誤出荷）と並ぶ運用品質の根幹工程。

---

## スコープ

- 一斉棚卸（全在庫を一度に数える）
- 循環棚卸（HT で日々ロケ単位に少しずつ）
- 差異発覚時の再カウント・確定フロー
- 棚卸履歴・トレース

## 機能リスト

### F-1101：一斉棚卸セッション開始

| 項目 | 内容 |
|------|------|
| 機能 | 棚卸対象（荷主×期間×範囲）を選んで「凍結」する |
| 副作用 | 対象範囲のロケを `locations.status = 'frozen'` に / 該当荷主の出荷指示を一時停止 |
| 出力 | `inventory_count_sessions` テーブルへの INSERT |
| 致命傷ライン | DB-4 / DB-5 |
| 9割突っ走り部分 | 開始ダイアログ UI |
| 完璧に詰める部分 | 凍結漏れ防止・出荷指示の一時停止判断・荷主への通知 |

### F-1102：HT 棚卸カウント

| 項目 | 内容 |
|------|------|
| 機能 | HT でロケ毎に在庫を数える |
| 入力 | バーコード読取（ロケ + SKU + 数量） |
| 出力 | `inventory_count_records` テーブルへの INSERT |
| 致命傷ライン | LK-2 / DB-2 / DB-3 |
| 9割突っ走り部分 | HT 画面の見た目 |
| 完璧に詰める部分 | 二重カウント防止・SKU 不一致時のリカバリー・部分カウント保存 |

### F-1103：差異検知

| 項目 | 内容 |
|------|------|
| 機能 | 帳簿在庫 vs 実在庫を照合し差異を検知 |
| ロジック | `inventory_count_records.actual_qty - inventory.quantity` |
| 出力 | `inventory_count_diffs` テーブル |
| 致命傷ライン | DB-1 / CA-2（差異額） |
| 9割突っ走り部分 | 差異一覧画面 |
| 完璧に詰める部分 | 差異の深刻度判定（許容差異 0.1% 以下 vs 重大差異） |

**差異額計算（CA-2 移動平均ベース）：**
```sql
SELECT
  d.sku_id, d.lot_id, d.location_id,
  d.actual_qty - d.book_qty AS diff_qty,
  ABS(d.actual_qty - d.book_qty) * i.unit_cost AS diff_value
FROM inventory_count_diffs d
JOIN inventory i ON i.id = d.inventory_id
WHERE d.session_id = ?;
```

### F-1104：再カウント・差異確定

| 項目 | 内容 |
|------|------|

## 棚卸方式の選択（オプション・荷主切替）

| 方式 | 特徴 | 推奨ターゲット |
|------|------|--------------|
| 一斉棚卸（年1回） | 全在庫を1日で数える・倉庫休止 | 規模小・SKU少 |
| 循環棚卸（毎日少しずつ） | 365日均等にロケを数える・倉庫運用継続 | 規模中〜大・SKU多 |
| ABC 重点棚卸 | A クラスは月次・B 半年・C 年次 | バランス型 |

**まーちゃん推奨：** 全方式を実装し、荷主マスタに「棚卸方式」フラグ。中小3PL は荷主によって方式が違う。

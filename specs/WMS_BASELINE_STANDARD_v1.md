# WMS 基本形（標準型）v1 — 製品の"型"定義

最終更新: 2026-06-08 / 策定: 4号 / 合意: 時吉さん（戦略壁打ち 2026-06-08）

## 目的
「どの倉庫でも、最初から機能する核」を **基本形（標準型）** として固定する。
多機能（481画面）は強みとして**残す**が、それは基本形の上の**拡張パック**であり、型ではない。
診断ウィザードが「基本形＝デフォルトON／拡張＝業種診断で追加ON」を実現する顔になる。

> 原則：**多機能 ≠ 売れる。** 売れる条件は「狭く・速く・確実に現場で回る核」。それが基本形。

---

## A. 基本形に含む「必須業務ループ」（これが回らなければWMSでない）

| # | 工程 | 必須機能 | 「機能する」の中身 |
|---|------|----------|------------------|
| 1 | マスタ | 商品 / ロケーション / 取引先 / 荷主(3PL時) / ユーザー | 登録・編集・**CSV取込**・検索。最小項目で立ち上がる |
| 2 | 入荷 | 入荷予定(ASN)登録/取込 → HT入荷検品(JAN・数量・ロット・期限) → 棚入れ → 在庫計上 | スキャン照合→在庫が実際に増える |
| 3 | 在庫 | 在庫照会(商品軸/ロケ軸)・ロット/期限(FEFO・該当商材のみ)・在庫調整(差異/理由) | 入出庫が即反映、ロケ単位で追える |
| 4 | 出荷 | 出荷指示登録/取込 → 引当(FEFO/FIFO) → ピッキングリスト → HTピッキング → 梱包 → 出荷確定 → 在庫引落 | 指示から在庫が実際に減るまで一気通貫 |
| 5 | 棚卸 | 棚卸表 → HTカウント → 差異確認 → 在庫調整反映 | 帳簿vs実数の差異が在庫に戻る |
| 6 | 帳票(最小) | ピッキングリスト / 棚卸表 / 納品書・送り状 / (3PL)月次請求 | PDF出力できる |

## B. 基本形に含む「運用の前提」（非機能・運用機能）
- **認証/ユーザー/権限ロール**、**荷主データ分離（3PL＝マルチテナント）**
- **データ投入**：CSV取込＋初期セットアップウィザード（現場が自走で立ち上げられる）← 導入の8割が死ぬ所
- **例外処理の基本**：欠品・数量差異・返品受付・保留(引当除外)。※現物が倉庫にある限り在庫から落とさない
- **監査ログ**（誰が・いつ・何を）、**HTオフライン最低耐性**、**レスポンス基準**（一覧/スキャンが体感即時）

## C. 「機能する」の受け入れ基準（型のDoD）
1. A・Bの各機能が**実データでend-to-end通る**（mock/注入でない＝backend実装が前提）
2. 上記ループを**初見の作業者が15分で一周**できる（time-to-value）
3. **例外（欠品/差異/返品）で止まらない**
4. 1荷主のデータが他荷主に**絶対に漏れない**（RLS実証）

## D. 拡張パック（＝型の外。残すが基本形ではない）
配送方面振分(Q) / 人員・シフト(K) / カレンダー(J) / 補充最適化(P) / 収支・KPIレポート(O) /
梱包材最適化(S詳細) / 外部倉庫・委託(M) / 競合調査由来の追加機能群 等
→ 診断ウィザードで業種に応じてON。基本形に**後付けできる**こと自体が設計要件。

## E. 役割分担
- **フロント / 画面 / 型定義 / デモ・オンボUX / 帳票 = 4号**
- **backend（DB・API・トランザクション・引当ロジック・RLS本番・性能）= 中津さん**
  - 製品化の本丸は **B/C の実データ化**。4号は基本形のAPI契約（発注書）を出す（→ [[API_CONTRACT_FRONTEND]]）。

## F. プロダクト化での位置づけ
- 基本形＝**デフォルトON**。拡張＝**診断で追加ON**。
- 診断デモは「基本形を全業種に即デプロイする顔」に育てる（mockデモ → 実プロビジョニング）。
- 売り方：全方位でなく**1セグメントに狭く刺す**（基本形が回る実績→事例→横展開）。

## 2. 横断する設計原則：いい機能を「使わせる」工夫

時吉さん指摘（2026-05-18）：WMS は良い機能が多いが「使いこなせない」印象。説明して「こんな機能あるから使ってね」と言っても使われない。**「使えないから要らない」ではなく、いい機能をいかに使わせるかを工夫する。**

### 2-1. 北極星（成功の指標）

**「気付かないうちに使っていた」** — 利用者が"最適化機能"を開いた覚えがないのに、作業が最適化されている状態。
さらに上の目標 = **他社 WMS を使うと"使いにくい"と感じるレベル**（時吉さん 2026-05-18）。
機能の良し悪しは「開かれた回数」でなく「気付かれず効いた回数」で測る。

### 2-2. 基本方針

- 機能を「独立した画面」でなく **「業務フローの中の提案」**にする。例：在庫配置最適化を別ページにせず、格納作業の画面でその場に「このSKUは B エリアが効率的」と出す。
- 利用者が「機能を探して使う」のではなく、**機能が業務の流れの中で向こうから出てくる**形に。
- 3スケールプロファイルで、その規模で使わない機能は見せない（情報過多を防ぐ）。
- シナリオテストで「どのシナリオでも自然に通らない機能」＝業務フローに埋め込まれていないサイン → 配置・出し方を見直す。

### 2-3. 「のっぺり問題」— 1ページ1機能の横伸ばし構造

時吉さん指摘（2026-05-18）：今の画面は「1ページごとに機能が1個ずつで、横にのっぺりしている」。
→ これは「画面数が300超で多い」と感じたのと **同じ原因の別症状**。1機能=1画面で横に伸ばした結果、
画面は浅く（のっぺり）／数は膨らみ（300超）／賢い機能は各々別ページに埋もれて見えない。

対策 = **横の広がり → 縦の密度**：
- フロー画面を「作業の拠点（ワークスペース）」にする。主作業を中心に、関連する提案・補助機能をその画面の中に重ねる。
- 利用者は1画面に留まり、画面の側が情報・提案を持ってくる（縦の深さ）。ページ間を横移動させない。
- 結果として **画面数はむしろ減る**（複数ページが1つの密な拠点に統合される）。
- 実際の画面統合は実装フェーズ寄りの大仕事。本ファイルでは原則のみ確定し、機能体系マップ見直し時に適用する。

### 2-4. 機能タイプ仕分け（A/B/C/D）

機能体系マップの全機能を4タイプにタグ付けし、「使われない機能」を炙り出す：

| タイプ | 内容 | 扱い |
|---|---|---|
| A | 画面の主役（探さず毎日開く基幹画面）| そのまま |
| B | 設定・マスタ（最初に1回、以後触らない）| そのまま |
| C | "提案"になるべき機能（今は独立画面、本来は業務フローの中に出すべき）| **要・出し方見直し** |
| D | レポート・分析（管理者が定期閲覧）| そのまま |

「使われない機能」はほぼ C に集中する想定。C のリスト = 「使わせる工夫バックログ」。
各 C 機能に「**どのフロー画面の、どの瞬間に、何のカードとして出すか**」を1行ずつ設計メモする。

### 2-5. 提案の絞り（うるさくしない）

「向こうから出す」をやりすぎると提案だらけで逆に使われなくなる。
- 提案は **常に1個だけ**（最優先のもの）。
- **判断が実際に変わるときだけ**出す（情報の確認だけなら出さない）。
- 3スケールが規模で機能を絞るのと同じ発想を、フロー画面の中でも適用する。

---

この原則は、新機能の採用判断・設計時、および機能体系マップの見直し時に必ず適用する。

### REQ-1 パレット管理

- パレットを **商品と同じように管理する**（パレットマスタ ＋ 所在・数量管理）。
- 区分：**自社所有**（固定資産的）/ **レンタル**（レンタル会社からの借受 — 返却義務・レンタル料あり）。
- 課題：**紛失が頻発する**。どこに何枚あるか、社外に出たまま戻らない分が見えない。
- 必要機能：
  - パレットの所在・数量管理（倉庫内ロケ／出荷で社外へ／回収待ち）
  - **貸出・返却の伝票起票機能**（パレットの出入りを伝票で記録）
  - レンタルパレットの返却管理・レンタル料
  - 紛失の記録、パレット棚卸
- IDEA-1 固定資産管理と一部重なるが、パレットは「動く・貸出返却される・紛失する」点で
  設備什器（動かない固定資産）とは別管理が要る → 独立した確定要件とする。
- 提案: 時吉さん（2026-05-18・確定）

### REQ-2 倉庫保管物管理（商品以外）

- 倉庫が **保管しなければならない「商品以外のもの」** を、ロケ＋保管期間で管理する。
- 代表例：**保管期間が定められた帳票・書類**（出荷伝票控え等、法定保存年限あり）。
- 課題：段ボールに入れたまま、**保管期間が過ぎても管理されず倉庫の隅に放置され続ける**。
- 必要機能：
  - 商品以外の保管物の登録・マスタ（帳票・書類 等）
  - ロケーション管理（どの段ボールがどこにあるか）
  - **保管期間管理**（保存年限の設定・期限の管理）
  - 期限超過の検知 → 「廃棄・処分可能」のアラート／リスト提示
- 設計原則 §2 と整合：期限が来たら向こうから「廃棄できます」と提案する形にする
  （隅に放置され続ける、をシステム側から防ぐ）。
- 提案: 時吉さん（2026-05-18・確定）

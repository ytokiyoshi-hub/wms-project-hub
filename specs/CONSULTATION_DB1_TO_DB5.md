## DB-1：在庫の持ち方

**論点：在庫レコードの管理粒度（どのキーの組み合わせで1行を特定するか）**

この設計が後から変わると、`inventory` テーブル全体の再設計が必要になります（+5〜8人日）。

### 選択肢の比較

| | A案 | B案 | C案 | D案 |
|---|---|---|---|---|
| 管理粒度 | SKU単位のみ | SKU × ロット | SKU × ロケーション | SKU × ロット × ロケーション × 荷主（4軸） |
| テーブルキー | `(owner_id, sku_id)` | `(owner_id, sku_id, lot_id)` | `(owner_id, sku_id, location_id)` | `(owner_id, sku_id, lot_id, location_id)` |
| データ行数 | 少ない | 中 | 中 | 多い |
| 実装難易度 | 易 | 中 | 中 | 難 |

### 各案の詳細

#### A案：SKU単位のみ
```
在庫レコード例：
owner_id=A, sku_id=商品X, quantity=100
（どの棚にあるか・どのロットかは不明）
```
- **メリット**：実装が単純。集計が速い
- **デメリット**：「棚Aの商品Xが50個、棚Bの商品Xが50個」の区別ができない。ロット追跡不可。現代のWMSとして機能不足

#### B案：SKU × ロット
```
在庫レコード例：
owner_id=A, sku_id=商品X, lot_id=ロット001, quantity=50
owner_id=A, sku_id=商品X, lot_id=ロット002, quantity=50
```
- **メリット**：製造日・期限切れ管理が可能。FIFO/FEFOの引当が可能
- **デメリット**：「どの棚にあるか」が不明。ロケーション機能が使えない

#### C案：SKU × ロケーション
```
在庫レコード例：
owner_id=A, sku_id=商品X, location_id=棚A-01, quantity=50
owner_id=A, sku_id=商品X, location_id=棚B-03, quantity=50
```
- **メリット**：棚の在庫が分かる。ピッキング最適化が可能
- **デメリット**：ロット追跡不可。食品・医薬品など期限管理が必要な荷主に対応できない

#### D案：4軸（SKU × ロット × ロケーション × 荷主）
```
在庫レコード例：
owner_id=A, sku_id=商品X, lot_id=ロット001, location_id=棚A-01, quantity=30
owner_id=A, sku_id=商品X, lot_id=ロット001, location_id=棚B-03, quantity=20
owner_id=A, sku_id=商品X, lot_id=ロット002, location_id=棚A-01, quantity=50
```
- **メリット**：「どの荷主の・どの商品の・どのロットが・どの棚に・何個あるか」を完全把握。FEFO引当・棚移動追跡・棚卸誤差分析すべてに対応
- **デメリット**：レコード数が増加（ただし中小3PLなら数十万行以内・パフォーマンス問題なし）。集計クエリが複雑になるため集計viewが必要

### 影響を受ける他機能

| 機能 | D案での実装方針 |
|------|--------------|
| 入荷登録 | ロット採番 + ロケーション指定で1レコード生成 |
| 棚卸 | ロケーション × ロット単位で差異を記録 |
| FEFO引当 | `expiry_date ASC, location_id ASC` でソート |
| ピッキング | 指定ロケーションから必要数を減算 |
| 棚移動 | 元ロケ減算 + 先ロケ加算（原子的トランザクション） |

### 開発チーム推奨：**D案（4軸）**

理由：
1. 中小3PLでも荷主×ロット×ロケでの管理は業界標準
2. 食品・医薬品荷主は期限管理（FEFO）が必須契約条件になる場合がある
3. 後からD案に移行する場合、全入荷・在庫データの再投入が必要（移行コスト大）
4. パフォーマンス懸念はview/index設計で対処可能

### 中津さんへの質問

1. **3PLの現場でA〜D案のどれが多いですか？** 中小3PLはB案で十分という判断もあり得ますか？
2. **荷主との契約でロット追跡を求められることはありますか？** 業種によって違いますか？
3. **D案でデータが増えた時のパフォーマンス問題**、経験則で「このくらいのデータ量から重くなる」という感覚はありますか？

## DB-2：ロット管理方式

**論点：ロット番号の粒度と付与方式（誰がどのタイミングでロット番号を発番するか）**

DB-1でD案（4軸）を選んだ場合、ロット番号の設計が在庫精度の鍵になります。

### 選択肢の比較

| | A案 | B案 | C案 | D案 |
|---|---|---|---|---|
| 方式 | ロット管理なし | 入荷ロット自動採番 | 製造ロット（メーカー指定） | 荷主ごとにB/Cを切替 |
| ロット番号の出所 | なし | システムが自動生成 | 荷主・メーカーが指定 | 荷主マスタのフラグで決定 |
| 実装難易度 | 不要 | 易 | 中 | 中〜難 |
| 柔軟性 | なし | 低 | 低 | 高 |

### 各案の詳細

#### A案：ロット管理なし
- DB-1でD案を選んでいる場合、`lot_id` を NULL 固定にする運用
- **現実的ではない**：ロット管理なしでは食品・医薬品荷主を受け入れられない

#### B案：入荷ロット自動採番
```
入荷時にシステムが自動採番：
lot_number = "INB-20260510-001"（入荷日 + 連番）
```
- **メリット**：オペレーターの入力作業なし。採番ルールが統一される
- **デメリット**：メーカー指定のロット番号が残せない。トレーサビリティが製造元まで遡れない

#### C案：製造ロット（メーカー指定番号をそのまま入力）
```
入荷時にオペレーターがスキャン or 手入力：
lot_number = "LOT-XYZ-2026-04-01"（メーカーが箱に印刷した番号）
```
- **メリット**：製造元まで遡れる完全なトレーサビリティ。リコール時に威力を発揮
- **デメリット**：オペレーターの入力作業が増加。荷主によって番号フォーマットが異なりバリデーションが複雑

#### D案：荷主ごとに方式を切替
```
荷主マスタ例：
A社（食品）: lot_method = 'manufacturer'  → C案（製造ロット入力）
B社（雑貨）: lot_method = 'inbound_batch' → B案（自動採番）
```
- **メリット**：荷主の業種・要件に合わせて最適な方式を選べる。1つのWMSで食品も雑貨も対応可能
- **デメリット**：荷主セットアップ時の設定が必要。ロット番号の意味がA社とB社で違うため、複数荷主一括レポート時に注意が必要

### 仕様への影響

```sql
-- lots テーブル構造（D案想定）
lots (
  id           bigint primary key,
  owner_id     bigint not null,  -- 荷主
  sku_id       bigint not null,  -- 商品
  lot_number   text not null,    -- A社はメーカー番号、B社はINB-日付-連番
  source_type  text not null,    -- 'manufacturer' | 'inbound_batch'
  mfg_date     date,             -- 製造日（製造ロットのみ）
  expiry_date  date,             -- 期限（FEFO引当に使用）
  received_at  timestamptz,      -- 入荷日時
  created_at   timestamptz
)
```

### 開発チーム推奨：**D案（荷主ごと切替）**

理由：
1. 中小3PLは荷主の業種が混在する（食品・雑貨・医療機器など）
2. SaaSとして「荷主ごとに機能ON/OFF」を実現する設計思想と整合
3. B案の自動採番をデフォルトにし、必要な荷主だけC案に切替 → 初期設定の手間を最小化

## DB-3：シリアル管理方式

**論点：1個1個の個体をシリアル番号で追跡するか（ロット管理の上位概念）**

シリアル管理はロット管理より細かい粒度です。「ロット001の商品Xが100個入荷した」ではなく、「商品X、シリアル番号SN-001からSN-100のどれが今どこにあるか」を追跡します。

### ロット管理 vs シリアル管理の違い

```
ロット管理：
  "2026-04-01製造ロットの商品Xが棚A-01に50個ある"

シリアル管理（ロット管理の上位互換）：
  "シリアル番号SN-001の商品Xが棚A-01にある"
  "シリアル番号SN-025の商品Xが棚B-03にある"
  → 1個1個の居場所・状態を全部追える
```

### 選択肢の比較

| | A案 | B案 | C案 | D案 |
|---|---|---|---|---|
| 方式 | シリアル管理なし | 対象SKUのみ（フラグ制御） | 全SKUシリアル管理 | 入荷時オペレーター判断 |
| 実装難易度 | 不要 | 中 | 高（データ量大） | 中（ルールが曖昧） |
| 運用コスト | 低 | 中（対象SKUのみ入力） | 高（全商品で個別スキャン） | 不定 |

### 各案の詳細

#### A案：シリアル管理なし
- 全商品をロット単位 or SKU単位でのみ管理
- **用途**：食品・雑貨など個体識別が不要な商品中心の倉庫
- **デメリット**：家電・医療機器・高額品を扱う荷主を受け入れられない

#### B案：SKUフラグで対象を限定
```
SKUマスタ例：
商品X（スマートフォン）: serial_required = true  → 個別スキャン必須
商品Y（ボールペン10本入り）: serial_required = false → ロット管理で十分
```
- **メリット**：必要な商品だけシリアル管理。データ量と運用コストを最小化
- **デメリット**：SKUマスタのセットアップ時に正確な設定が必要。設定漏れのリスク

#### C案：全SKUシリアル管理
- 全商品に個別シリアル番号を割り当て
- **メリット**：完全なトレーサビリティ。紛失・誤出荷を1個単位で追跡
- **デメリット**：入荷・検品・ピッキングすべてで1個ずつスキャン → 大幅な作業時間増加。中小3PLには非現実的

#### D案：入荷時オペレーター判断
- 入荷担当者がその都度シリアル管理するか決める
- **デメリット**：ルールが属人化。入荷ごとにブレが生じる → 在庫不整合の原因になりやすい

### 仕様への影響

```sql
-- serials テーブル（B案想定）
serials (
  id            bigint primary key,
  owner_id      bigint not null,
  sku_id        bigint not null,  -- serial_required = true のSKUのみ
  lot_id        bigint,           -- ロットと紐付け（任意）
  serial_number text not null,
  status        text not null,    -- 'in_stock' | 'picked' | 'shipped' | 'returned' | 'damaged'
  location_id   bigint,           -- 現在の棚位置
  received_at   timestamptz,
  shipped_at    timestamptz
)

-- skus テーブルへの追加
ALTER TABLE skus ADD COLUMN serial_required boolean DEFAULT false;
```

### HTバーコードとの連動

HTバーコード（致命傷ラインLK-2）との連動により：
- `serial_required = true` の商品は、入荷・検品・ピッキング時に個別スキャンを**システムが強制**
- スキャンなしでは次の工程に進めないUI制御

### 開発チーム推奨：**B案（SKUフラグ制御）**

理由：
1. 全SKU一律（C案）は中小3PLの現場では非現実的（作業時間が倍以上になる）
2. 対象を絞ることで、シリアル管理が本当に必要な高額品・家電だけに集中できる
3. HTバーコードとフラグ連動で「設定した商品は確実にスキャンする」仕組みが作れる
4. 将来的に`serial_required`をONにするだけで、既存商品にも適用可能（拡張性あり）

2. **B案のフラグ制御アプローチ**、3PLの現場として現実的ですか？

## DB-5：ロケーション管理方式

**論点：倉庫内の棚（ロケーション）をどのような階層・種別で管理するか**

この設計はピッキング効率・棚卸精度・引当ロジックすべてに直結します。
後から変えると、ロケーション関連の全APIとHTバーコード読取ロジックの再設計が必要（+5〜7人日）。

### 倉庫のロケーション構造（イメージ）

```
倉庫
├── エリアA（冷蔵・温度管理品）
│   ├── 通路A01
│   │   ├── 棚A01-1（ピッキング用・段1）
│   │   └── 棚A01-2（ピッキング用・段2）
│   └── 通路A02
├── エリアB（一般品・保管棚）
│   ├── 通路B01
│   └── 通路B02
└── 仮置きエリア（入荷後の一時保管・不良品置き場）
```

### 選択肢の比較

| | A案 | B案 | C案 | D案 |
|---|---|---|---|---|
| 方式 | フラット（コードのみ） | 階層化（エリア > 通路 > 棚 > 段） | 機能種別（保管/ピッキング/仮置き/不良品） | 階層 × 機能種別（B+Cハイブリッド） |
| ロケーションコード例 | `A-01-01` | `エリアA / 通路A01 / 棚1 / 段1` | `storage` `picking` `staging` | `エリアA / 通路A01 / 棚1 / 段1 / type=picking` |
| 実装難易度 | 易 | 中 | 中 | 難 |
| ピッキング最適化 | 不可 | 可（通路順に並び替え） | 可（種別で分類） | 完全対応 |

### 各案の詳細

#### A案：フラット（ロケーションコードのみ）
```
locations テーブル：
id=1, code="A-01-01", quantity=100
id=2, code="A-01-02", quantity=50
```
- **メリット**：シンプル。初期実装が速い
- **デメリット**：ピッキング時の棚順ソートができない。ABC分析による引当優先度制御が不可。将来的な機能拡張が困難

#### B案：階層化
```
locations テーブル：
id=1, area="A", aisle="A01", rack=1, level=1, code="A-A01-1-1"
id=2, area="A", aisle="A01", rack=1, level=2, code="A-A01-1-2"
```
- **メリット**：通路順にピッキングリスト並び替え可能。エリア別在庫レポートが出せる
- **デメリット**：ロケーションの機能種別（保管用か仮置き用か）が区別できない

#### C案：機能種別
```
location_type:
  storage  → 通常保管棚
  picking  → ピッキング専用棚（少量・取り出しやすい位置）
  staging  → 入出荷前の仮置き
  damage   → 不良品置き場
  returns  → 返品品置き場
  inspection → 検品エリア
```
- **メリット**：ロケーションの役割が明確。仮置きエリアを在庫から除外した集計が容易
- **デメリット**：棚の位置情報がないため通路順ピッキングが不可

#### D案：階層 × 機能種別（B+Cハイブリッド）
```sql
-- locations テーブル構造
locations (
  id            bigint primary key,
  owner_id      bigint,              -- 荷主専用ロケ or NULL（共用）
  code          text not null,       -- "A-A01-1-1"
  area          text,                -- エリア（A/B/C/冷蔵...）
  aisle         text,                -- 通路番号
  rack          integer,             -- 棚番号
  level         integer,             -- 段
  location_type text not null,       -- storage/picking/staging/damage/returns/inspection
  abc_class     text,                -- A/B/C（ABC分析クラス）
  picking_priority integer,          -- ピッキング順
  temp_zone     text                 -- normal/refrigerated/frozen
)
```

- **メリット**：
  - 通路順 × 段順のピッキングリスト最適化
  - ABC分析クラスを自動引当に活用（Aクラス商品はピッキング棚に常駐）
  - 機能種別で「在庫」と「一時保管」を明確に分離
  - 荷主専用ロケーション（`owner_id`指定）と共用ロケーションを混在可能
- **デメリット**：初期の棚マスタ入力が多い（ただし一度設定すれば変更不要）

### ピッキング最適化の具体例

```sql
-- ピッキングリストの生成（D案）
SELECT l.area, l.aisle, l.rack, l.level, l.code, i.quantity
FROM inventory i
JOIN locations l ON i.location_id = l.id
WHERE i.owner_id = :owner_id
  AND i.sku_id IN (:target_skus)
  AND l.location_type IN ('storage', 'picking')
ORDER BY l.aisle, l.rack, l.level;
-- → 通路順・棚順にソートされたピッキングリストが生成
```

### ABC分析との連動

```
Aクラス商品（出荷頻度高）→ picking ロケ（取り出しやすい前列・下段）
Bクラス商品（出荷頻度中）→ storage ロケ（中段）
Cクラス商品（出荷頻度低）→ storage ロケ（高段・奥）
```

引当時に `abc_class = 'A'` かつ `location_type = 'picking'` を優先することで、ピッキング効率を最大化できます。

### 開発チーム推奨：**D案（階層 × 機能種別）**

理由：
1. 中小3PLでもこの粒度はほぼ必須（現場のオペレーション効率に直結）
2. ピッキングリストの通路順ソートは「WMSとして当然の機能」として期待される
3. A案から始めると、将来D案に移行する際に全ロケーションデータの再設計が必要
4. HTバーコード（致命傷ラインLK-2）でロケコードをスキャンする際、階層情報があると検証が容易

## 4項目まとめ・開発への影響

| 論点 | 開発チーム推奨 | 変更リスク（後から変えた場合） |
|------|--------------|---------------------------|
| DB-1：在庫の持ち方 | **D案（4軸）** | inventoryテーブル全再設計（+5〜8人日） |
| DB-2：ロット管理 | **D案（荷主切替）** | ロット関連テーブル・API全修正（+3〜5人日） |
| DB-3：シリアル管理 | **B案（SKUフラグ）** | serialsテーブル追加・HTロジック修正（+2〜3人日） |
| DB-5：ロケーション管理 | **D案（階層×機能種別）** | locationsテーブル全再設計・引当ロジック修正（+5〜7人日） |

### 4項目の相互依存関係

```
DB-1（4軸在庫） ──依存──→ DB-5（ロケーション管理）
      │                         │
      └───依存──→ DB-2（ロット管理）──→ DB-3（シリアル管理）

※ DB-1がD案なら、DB-2・DB-5もD案推奨の設計と整合
※ DB-3（シリアル）はDB-2（ロット）の上位概念として追加実装
```

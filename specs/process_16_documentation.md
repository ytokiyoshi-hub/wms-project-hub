| 選択肢A | **引き継ぎ書最優先**：Phase 9 実装が進む前に技術的引き継ぎ書を整備。設計判断を鮮度の高いうちに記録 |

## Q2：ユーザー向けマニュアルの対象と形式

| 項目 | 内容 |
|------|------|
| 論点 | ユーザー向けマニュアルの「読者」と「形式」をどう定めるか |
| 選択肢A | **倉庫作業者向け**：入荷・検品・ピッキング・出荷の操作手順（スクリーンショット付き・HT 操作含む）|
| 選択肢B | **荷主向け**：在庫照会・発注登録・請求明細確認の操作手順（荷主ポータル画面中心）|
| 選択肢C | **管理者向け**：SKU マスタ登録・ロケーション設定・ユーザー権限付与の手順 |
| 選択肢D | **3者すべて別冊**：A + B + C を対象者別に分冊して作成 |
| こーちゃん推奨 | **D（3者別冊）** |
| 推奨理由 | 倉庫作業者・荷主・管理者では見る画面も操作内容も全く異なる。1冊にまとめると「自分に関係ない手順」が多くなり現場で使われなくなる。分冊にして「自分が使う冊子だけ渡す」運用が最も現場フレンドリー |
| 時吉さん回答 | **D（確定 2026-05-17）：ユーザーマニュアルは3種類別冊（作業者向け・荷主向け・管理者向け）。** |

### D案 詳細：対象者別マニュアルの構成

| 冊子 | 主な対象 | 必須カバレッジ | 優先度 |
|------|---------|--------------|--------|
| **作業者向け** | 倉庫スタッフ | 入荷・検品・入庫・ピッキング・出荷・返品・HT 操作手順 | ★★★ |
| **荷主向け** | 取引先 EC・メーカー | 在庫照会・発注登録・請求明細確認・問い合わせフロー | ★★ |
| **管理者向け** | 3PL 内部の管理担当 | SKU 登録・荷主追加・権限管理・月次締め処理 | ★★★ |

## Q5：ドキュメント更新・メンテナンス体制

| 項目 | 内容 |
|------|------|
| 論点 | 実装後に画面・仕様が変わったとき、誰がどのタイミングでドキュメントを更新するか |
| 選択肢A | **こーちゃん責任**：実装変更時に自分でドキュメントも同一 PR で更新 |
| 選択肢B | **わーちゃん責任**：月次でわーちゃんがドキュメント全体を巡回レビュー・更新 |
| 選択肢C | **CI 強制**：docs/ 配下のファイルを変更しないと PR がマージできない仕組みを CI で入れる |
| 選択肢D | **A ＋ B**：こーちゃんが実装時に同時更新（一次責任）＋ わーちゃんが月次で読みやすさ・抜けを確認（二次責任）|
| こーちゃん推奨 | **D** |
| 推奨理由 | 失敗パターン6（自分のメモ運用してない）の教訓を踏まえ、「書く責任（こーちゃん）」と「確認・整形の責任（わーちゃん）」を分離する。CI 強制（C案）は PR ごとにドキュメント変更を強制するため、小さなバグ修正でも docs を触る必要が生じてコストが高い |
| 時吉さん回答 | **D（確定 2026-05-17）：実装担当者が一次責任（実装変更時に同時更新）＋わーちゃんが月次巡回で二次確認。Q16-1 同期更新原則（変更あれば即更新）と整合。** |

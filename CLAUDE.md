## ⚠️ 最重要：作業の進め方

### 確認なしで自律的に進める

このプロジェクトでは、**Claude Code に確認を求めずに自律的に進めてほしい**という方針です。

**やってほしいこと**：
- 指示された作業を最後まで実行する
- 細かい確認（「これでいいですか？」等）はしない
- 軽微な判断は自分で決定して進む
- エラーが出ても可能な限り自己解決
- 完了したらまとめて報告

**やらないでほしいこと**：
- 作業途中で確認を求めて止まる
- 「次は○○しますか？」のような確認
- 些細な選択肢を毎回聞く

### 自己判断でOKな範囲

- ファイル名・パス・命名の細部

- 既存の確定事項を覆す判断

それ以外は、**判断して進む** ことを優先してください。

## プロジェクト概要

### 名称
**WMS Project Hub** - 倉庫管理システム（3PL対応版）の仕様策定・開発進捗管理プロジェクト

### 対象システム
- **WMS（Warehouse Management System）3PL対応版**
- 3PL：複数荷主受託型物流（倉庫事業者が複数の荷主の在庫を管理する形態）
- ベースシナリオ：株式会社マルキ食品（食品卸 BtoB+EC、**架空企業**）
- HT機種：キーエンス BT-A2000（4インチ）/ BT-A1000（3インチ）共通設計

### このプロジェクトの位置づけ
- 現在は**仕様策定フェーズ**（実装前）
- お客様（物流のプロ・非エンジニア）と Claude（別チャットセッション）が議論を進めている
- 確定した仕様を、このリポジトリに反映するのが Claude Code の役割

### 公開サイト
**https://ytokiyoshi-hub.github.io/wms-project-hub/**

## 重要な業務ルール

### マスタコード体系（統一）

すべて「英2桁＋3桁数字」で統一：

| マスタ | 形式 | 例 |
|---|---|---|
| 倉庫（自社） | WH___ | WH001 |
| 倉庫（外部委託） | YU___, TK___ 等 | YU001 |
| 荷主 | MK___ | MK001 |
| ピッキングエリア | AR___ | AR001 |
| 配送業者 | CR___ | CR001 |
| 取引先 | TP___ | TP001 |

### 枝番管理

商品マスタ・取引先マスタ・荷主マスタで採用：
- 通常は枝番なし（例：S-00321、TP001）
- 必要時のみ-02から追加（例：S-00321-02、TP001-02）
- 履歴重視（リネームせず保持）

### ロケーションコード

```
{階}{フロア区分}{種別英字}-{通路2桁}-{列}{段}{位置}
例：3AP-02-134（3階Aフロア・パレットロケ-02通路-1列3段4位置）
```

種別英字（顧客カスタマイズ可）：
- P: パレット
- T: 棚（バラ）
- F: 流動棚
- K: 仮置き
- R: 不適合品
- H: ハイラック

### 用語の統一

倉庫業界の正式用語を使用：

| 誤った表現 | 正しい表現 |
|---|---|
| 受注 | 出荷／出荷指示 |
| 課金 | 請求／料金（文脈による） |
| もらう／もらわない | 収受／控除 |

「課金」は「請求管理」「料金計算」等、文脈で使い分け。

### 単位構造（4階層）

商品の単位：
- パレット
- ケース
- ボール
- ピース

商品マスタには「1段下の単位」で持つ（階層関係を明示）。

### 計算方式（保管料）

| 方式 | 計算 |
|---|---|
| 3期制 | (前月繰越残 + 10日残 + 20日残 + 入庫総数) × 単価 |
| 2期制 | (前月繰越残 + 15日残 + 入庫総数) × 単価 |
| 坪貸し | 確保面積 × 単価（固定） |

期分けは「締め日＋期制」から自動算出。

## 廃止された設計（注意）

以下は誤った概念だったため**廃止確定**。実装してはならない：

- W-310 その他出庫（PC画面）
- HT-801 その他出庫（HT画面）

理由：倉庫事業者は荷主からの寄託物を保管している立場で、倉庫事業者の意思で勝手に処分することは法律・契約上不可。「廃棄」「サンプル」「仕入先返品」も全て**荷主の指示として「受注」で処理**する。

仕様策定段階の画面モックは、**機能・画面構成の確認用**として簡素なデザインで統一：

- カラー：紺（#1a3a5c）・赤（#b8002e）・グレー
- フォント：Noto Sans JP + Roboto Mono
- 装飾は最小限
- レイアウトの一貫性を優先

実装フェーズでは別途、洗練されたデザイン（参考画像あり）を採用予定。

### Hub のデザイン（index.html）

- 管理ツール風（GitHub風）
- ダーク基調のサイドバー
- 明るい背景のメインエリア
- ステータス色分け（完了:緑 / 進行中:青 / 未着手:グレー / 警告:橙）

## Git の使い方

### コミットメッセージのルール

- 簡潔な日本語で内容を表現
- 例：「Hub更新 - 工程1完了反映」「画面モック追加 - 商品マスタ」「READMEのリンク修正」

### ブランチ運用

- 通常は main ブランチに直接コミット
- 大きな変更時のみフィーチャーブランチを切る判断はClaude Codeに委ねる

### Push の頻度

- 作業がひと区切りついたらすぐ push
- 確認は不要

## 困った時

### 不明点が出た場合

CLAUDE.md やリポジトリ内のドキュメントを参照して判断。
それでも不明な場合のみ、お客様に質問する。

### エラーが出た場合

- まず自己解決を試みる
- 解決不可なら、エラー内容と試した対処を明記して報告

## 最終確認

このプロジェクトでの Claude Code の役割：

1. **自律的に作業を進める**（確認なしで判断）
2. **仕様を理解した上で正確に実装**（CLAUDE.mdの内容を遵守）
3. **完了したら Git でcommit & push**（自動化）
4. **完了報告は最後にまとめて**（途中報告はしない）

これらが守られれば、お客様との対話セッション（別チャット）が議論に集中でき、Claude Code が実装を担当する効率的な分業が成立します。

## まーちゃん行動ルール（絶対守る）

### 1. 指示出し専任（failure_pattern #5）
- Edit/Write/Bash でファイル修正・実装は禁止
- 全てタスク起票→委任。自分ではコードを書かない

### 2. 確認・検証・見直し3点セット（failure_pattern #2/#4/#26）
- 完了報告前に必ずログ/DBで実態確認
- 「コード書いた」≠「完了」。動作確認してから完了
- 「全員」「全件」と書く時はgrepで母集団確認

### 3. こーちゃんは急ぎ専任（failure_pattern #24）
- 通常タスクはさーちゃん(5)・にーちゃん(7)へ
- こーちゃんのキューは常に空が正常

### 4. 反省ループ禁止（failure_pattern #7）
- 「反省します」「徹底します」で終わらせない
- 即タスク起票→仕組み化に変換する

### 5. 1問ずつ（failure_pattern #3）
- 時吉さんへの質問・依頼は1メッセージ1件のみ

### 6. DB操作の安全ルール（failure_pattern #16）
- 複数行を一括でstatus変更しない
- approved投入は1件ずつシリアルに

### 7. 完成後動作検証（failure_pattern #26）
- 定期実行系: 実際に発火したログを確認
- LINE通知系: 実際にLINEに届いたか確認
- Skill系: TRIGGERで実際に発動するか確認

## 全タスク標準フロー（必須）

時吉さん指示：全タスクをこのフローに統一する。

### Step 1: 起票
- タスクをpendingでINSERT → まーちゃんがapprovedに変更
- descriptionに完了条件・テスト手順を明記してから起票

### Step 2: 実装（作業）
- in_progressで作業開始
- 実装完了してもまだcompletedにしない

### Step 3: テスト（動作確認）
- 実装した機能が実際に動くかをログ・DB・実機で確認
- Skillを実装した場合: TRIGGERキーワードで実際に発動するかテスト
- 定期実行系: 実際に発火したログを確認
- LINE系: 実際にLINEに届いたか確認
- テスト結果をkurokun_memoに記録（category=test_result）

### Step 4: 動作確認OK報告
- 全テスト通過後のみcompleted
- completedにする前に task-completion-verify Skill を発動
- 「動作確認OK: [確認した内容]」をログに記録
- まーちゃんへ報告（kurokun_outboxまたはSELECT結果）

### 違反した場合
- テストせずcompletedにした = failure_pattern #26再発
- 「動作確認OK報告」なしの完了報告は受け付けない

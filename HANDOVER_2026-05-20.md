### CSS クラス（test2.css）

| クラス | 役割 |
|---|---|
| `.two-pane` | 2ペイン grid（minmax(0,1fr) 320px / align-items:stretch） |

| `.card.fit` | カードを中身幅にフィット（layout_c時代の名残・現在は基本未使用） |

### テーブル列幅型 (`table.list.list-full`)

- `width: 100%; table-layout: auto`（fixed から auto に変更済み・列幅暴れ防止）
- c-XXX クラスで列タイプ別の width 指定
- `c-code` 120px / `c-num` 80px / `c-date` 160px / `c-status` 80px / `c-action` 100px / `c-text` 320px（max-width:none）
- 特殊画面（products.html等6列）は inline `style="width:180px"` で c-text を画面別調整

### th 揃え型

- **th はデータ揃えに追従**（c-num→右 / c-code/c-text/c-date→左 / c-status/c-action→中央）
- ヘッダの中央がデータの中央に揃う原則
- 経緯: 一度「全左→全中央→データ中央」と試行し、最終的に「th はデータに合わせる」で確定

### フィルタ型 (`.filter-row`)

- 検索フィルタ: `display: flex; gap: 14px; flex-wrap: wrap; align-items: flex-end`
- `.field input/select { width: 200px }` 固定
- 登録/編集フォーム (`.form-row`) とは別物・混ぜない
- **ボタン主導**: 即時検索（addEventListener）は削除済み・検索ボタン押下のみで render

### 通知型（`window.alert` 上書き → トースト）

- `clToast(msg, kind, durationMs)`
- kind: `info`（3.5秒）/ `success` / `important`（消えない）/ `error`（消えない・赤）
- `window.alert` 上書きで既存全コードを自動トースト化
- `clConfirm(msg, opts)` styled モーダル（共通関数のみ用意・置換は後続）

### キャッシュバスティング

- 全 HTML の `<link href="/css/test2.css?v=YYYYMMDDHHMM">` でバージョン管理
- **CSS 変更時は ?v= 必ず bump**（怠るとブラウザに反映されない・1日この問題で混乱した）

### ヘルプアイコン（説明文ブロック）

- 画面タイトル横に `<button class="intro-help" data-intro="..." data-intro-code="A-9">` を1個
- クリックでポップオーバー（fade+slide 140ms）
- 共通 `bindIntroHelp()` で動的にポップオーバー DOM 生成

### バッジ（`.pill`）

- `min-width:60px; padding:2px 10px; text-align:center` で文字数差を吸収
- table の c-status 列内では `display:block; min-width:0` で 100% 幅

## 3. 1号↔2号 チャンネル運用

### 仕組み

- **ディレクトリ**: `~/github/wms-impl/`
- **1号→2号**: `MESSAGE_1_TO_2_{タスク名}_{日付}.md` を Write
- **2号→1号**: `MESSAGE_2_TO_1_{タスク名}_DONE_{日付}.md` を Write
- **2号 watcher**: PID 63361（zsh while-loop・20秒ごとに新MESSAGE検知・自動読み込み）

### 1号側の監視（重要）

2号は MESSAGE_2_TO_1 報告を**まとめて出す癖**があり、コミットだけ続けて報告が滞ることがある。
従って **git log 新コミット即検知** で監視する：

```bash
LAST_HASH=$(git -C ~/github/wms-test2 log -1 --format=%H)
i=0
until [ "$(git -C ~/github/wms-test2 log -1 --format=%H)" != "$LAST_HASH" ] || [ $i -ge 40 ]; do
  sleep 30; i=$((i+1))
done
echo "EVENT=NEW_COMMIT iter=$i"; git -C ~/github/wms-test2 log -1 --format='%h %s'
exit 0
```

- `exit 0` 必須（最後の式が偽だと task が failed 表示になる）
- 上限 60〜120 iter（30〜60分）

### 2号は wms-test2 / wms-impl の実装担当

- wms-test2 リポジトリの編集権限を持つ（1号は読み取り・指示出し）
- wms-project-hub は2号は読み取り専用（仕様編集・commit は1号のみ）

## 4. verify-before-report 原則（絶対）

**1日で2回連続 verify ミスをして時吉さんを混乱させた**。今後の鉄則：

1. コミットメッセージを**鵜呑みにしない**（実 HTML を grep で確認）
2. grep の判定ロジックは**厳密に**（`<tbody id="tb">` の属性を「生タグ露出」と誤判定した）
3. 「完了」と書く前に **実 HTML を curl + grep** で：
   - `?v=` が bump されてるか
   - 該当クラス（list-full / filter-row / two-pane 等）が付いてるか
   - 生タグ露出（`grep -E '^[[:space:]]*id='` でタグ外テキスト）

### 整合性チェッカー（tools/）

- `tools/check-th-td-integrity.mjs` ← th と td の c-XXX 整合性チェック（不一致0件維持）
